# Superpowers 根约束 - Root Discipline

**版本**: v1.0
**创建日期**: 2026-05-15
**状态**: 生效中
**来源**: `CLAUDE.md §1~§6`、`.cursor/rules/superpowers-and-context.mdc`

> 本文件是 `.kiro/steering/` 的**根节点**。所有其他 steering 文件是它的具化。spc-planning.md 是它的执行层翻译。任何 steering 文件与本文件冲突时，以本文件为准。

## §1 五步顺序（不可跳、不可逆）

```
Step 1: 固定目标/商业定位
Step 2: 固定角色与职责
Step 3: 固定通信边界
Step 4: 固定任务与能力模型
Step 5: 最后才进入实现
```

违反后果：跳过前四步直接写代码 → 架构崩溃。SPC 模式的 Spec→Plan→Code 是本五步顺序的每轮迭代执行方式。

## §2 六条补充纪律

| # | 纪律 | 含义 |
|---|------|------|
| 1 | 不猜 | 任何不确定的前置条件必须先验证或先问用户，不允许"猜了就写" |
| 2 | 先对齐 | 基准（数据表、硬件文档、接口定义）出现分歧时先对齐再动代码 |
| 3 | 先告知 | 发现"基准错了"必须先停下告知用户，不可默默修正 |
| 4 | 先验证 | 凡能用 schema / 单测 / 仿真器验证的，都前移到设计期 |
| 5 | 不一大块 | 拒绝大块一次性落代码，按里程碑分阶段，每段独立 commit |
| 6 | 先技能 | 只要任务可能与某 skill 相关，先加载并对照执行 |

## §3 上下文过半强制换窗

当会话上下文达到约 50% 时**必须**：

1. 提示用户新开窗口
2. 压缩前把"本轮决议、待办、约束"落到仓库文档或 task 列表
3. 不允许"快满了还继续塞"
4. 压缩后第一句必须重新确认当前里程碑和下一步

### §3bis 记忆混乱自检

出现以下任一征兆即视为记忆开始混乱：

- 反复确认已经确认过的事
- 已明确的基准被复问或被改错
- "之前我们说过"与实际记录对不上
- 跨工具调用之间丢失上下文
- 建议与先前定稿的架构/约束冲突且无新依据
- 压缩过一次后仍不断触发类似混乱

必须立即告知用户："当前上下文已出现记忆混乱征兆，建议新开窗口并从 XX 里程碑继续"。

新窗接续必须：
1. 落"当前状态快照"（里程碑、已完成项、下一步、待拍板问题）
2. 给一句可复制的接续指令
3. 不依赖上下文摘要的记忆硬撑

## §4 跨项目通用约束

- 永不推送含 `.claude/`、`.env`、凭据、密钥的 commit
- 公开仓库操作（push / PR / gist）先明确用户意图
- 破坏性 git 操作（force push / reset --hard / branch -D）必须用户明确请求
- 第三方库升级单独 commit，不混业务代码

## §5 编码后文档同步（强制）

命中任一阶段性节点后立即同步文档（不可延迟）：

| 节点 | 必须更新的文档 |
|------|--------------|
| 里程碑最后一项完成 | `架构定稿` + `实施计划` + `M0进度报告` + `全局规划` |
| Capability 端到端跑通 | `架构定稿` + `实施计划` |
| 芯片/接口/边界核对 | `硬件核对报告` + `硬件连接说明` + 对应固件 config |
| 基准修订 | `架构定稿` + 修订记录 |
| Commit 对应逻辑闭合 | 相关文档 |

更新顺序：先验证 → 立即更新文档 → 文档与代码一并 commit → 文档没更新前下一个节点不开始。

例外（不触发同步）：纯 typo、注释修复、格式化、完全内部重构（行为不变、接口不变、文档无感知）。

## §6 Planning with Files — 文件系统即工作记忆

Context Window = RAM（挥发有限），Filesystem = Disk（持久无限）。

### 6.1 触发条件

任务涉及 3+ 步骤 / 跨多个 tool call / 涉及研究探索构建 / 用户说"开始做 X 功能" → 立即创建计划文件。

跳过：单文件编辑、简单问答、快速查找。

### 6.2 三类必需文件

| 文件 | 用途 | 更新时机 |
|------|------|----------|
| `task_plan.md` | 阶段、进度、决策 | 每个阶段完成后 |
| `findings.md` | 研究发现、接口发现 | 发现任何东西后立刻 |
| `progress.md` | 会话日志、测试结果 | 持续更新 |

### 6.3 核心规则

1. 先建计划再动手：无 `task_plan.md` 不得开始复杂任务
2. 2 动作规则：每 2 次查看/搜索后立即写 findings.md
3. 决策前先读 plan 刷新目标
4. 行动后立即更新：`in_progress` → `complete`，记录产物
5. 记录所有错误：Error、Attempt、Resolution
6. 禁止重复失败：`if action_failed: next_action != same_action`

### 6.4 三击错误协议

```
尝试 1: 诊断 → 修复（读错误信息，定位根因，定点修复）
尝试 2: 换思路 → 换方法/工具/库（绝不重复相同失败动作）
尝试 3: 重新审视 → 挑战假设，搜索方案，考虑更新计划
3 次失败后: 升级给用户（说明尝试内容 + 具体错误 + 请求指导）
```

### 6.5 读/写决策矩阵

| 场景 | 动作 | 理由 |
|------|------|------|
| 刚写完文件 | 不读 | 内容仍在上下文中 |
| 查看了图片/PDF | 立即写 findings | 多模态信息丢失前写入文本 |
| 开始新阶段 | 读 plan + findings | 上下文可能已过期 |
| 出现错误 | 读相关文件 | 需要当前状态来修复 |
| 中断后恢复 | 读所有计划文件 | 恢复状态 |
| 上下文压缩时 | 先写入所有决议再压 | 压缩后接续依据 |

## §7 Windows 环境约束

- 路径分隔符优先用 `/`，Windowns 原生程序用 `\`
- 不用 `find`、`xargs -0`、`ls /dev/...` 等 Unix 命令，用 Glob/Grep 等内置工具
- `/dev/null` 可用（Git Bash），不用 `NUL`
- 含空格路径加双引号
- PDF/二进制用 Python + PyMuPDF，不假设 `pdftoppm`、`pdftotext` 存在
- `cd` 能不用就不用
