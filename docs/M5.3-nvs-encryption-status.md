# M5.3 NVS encryption status

## Scope

This slice adds the design-time firmware contract for encrypted storage of WiFi credentials and device secrets on the U8 Xiaozhi firmware.

## Implemented

- `firmware/u8-xiaozhi/sdkconfig.defaults` enables `CONFIG_NVS_ENCRYPTION=y`.
- `firmware/u8-xiaozhi/partitions/v2/16m.csv` reserves an encrypted `data,nvs_keys` partition at `0xd000`.
- `ota_0` remains fixed at `0x20000` so the existing OTA app layout is not shifted.
- `firmware/u8-xiaozhi/main/main.cc` now initializes NVS through `InitNvsStorage()`.
- When `CONFIG_NVS_ENCRYPTION` is enabled, startup looks up the `nvs_keys` partition, reads or generates NVS encryption keys, then calls `nvs_flash_secure_init()`.
- When `CONFIG_NVS_ENCRYPTION` is not enabled, startup keeps the previous `nvs_flash_init()` path.
- The existing NVS erase/retry path remains for truncated or newer-version NVS partitions.
- Static CI coverage checks the default config, partition table, secure-init calls, fallback init path, and credential key contract.

## Credential keys covered by the contract

- Namespace: `wifi`
- Keys: `ssid`, `password`, `device_secret`, `server_host`

## Not yet validated

- ESP-IDF build for the U8 target with this exact sdkconfig resolution.
- Real-device first boot key generation in `nvs_keys`.
- Reboot persistence after writing WiFi credentials and `device_secret`.
- Flash dump inspection proving plaintext credentials are not present.
- Final production security posture: ESP-IDF 5.5 requires either flash encryption for `nvs_keys` protection or an HMAC/eFuse key-protection scheme. This slice adds the firmware contract but does not provision eFuse keys or enable production flash encryption.

## NVS Encryption Evidence Gap Record

Use this structure when encrypted credential-storage evidence is missing for a release candidate. It is not evidence of production NVS key protection.

- missing evidence scope: resolved ESP-IDF sdkconfig, first-boot `nvs_keys` generation, credential write, reboot persistence, flash dump inspection, flash encryption posture, HMAC/eFuse key-protection posture, or production key-provisioning procedure
- environment: staging device, release-candidate firmware, ESP-IDF version, partition table, and sdkconfig artifact
- owner: named firmware or security owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep credentials treated as not production-proven, avoid storing production secrets on unverified firmware, and block release if plaintext credentials are found or key-protection posture is unresolved
- risk acceptance: firmware owner, security owner, and release manager approval reference
- rollback trigger: sdkconfig disables NVS encryption, `nvs_keys` is missing, reboot loses credentials, flash dump contains plaintext credential material, or production eFuse/flash-encryption posture is not approved
- follow-up evidence: resolved sdkconfig, partition table, first-boot serial log, credential write log, reboot verification log, redacted flash dump inspection output, eFuse or flash-encryption posture approval, and key-provisioning record
