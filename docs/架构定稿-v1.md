# esp32S_XYZ 架构定稿 v1

> **状态：历史快照。现行版本为 [架构定稿-v2.md](./架构定稿-v2.md)。**
>
> v1 仅覆盖 `Server / U8 / U1` 三层，未包含客户端、业务分层与商业化约束。
> v2 是 v1 的超集：保留 v1 已落到代码的所有条款（设备 capability、U8↔U1 私有协议、U1 状态机与错误码），并在其之外补全 `Client / BusinessServer / DeviceServer` 角色、通信边界与资源/归属模型。
> 继续使用 v1 仅用于查阅设备层细节；商业化、客户端、订阅、投影管线等决策以 v2 为准。

本文档只记录已经确认的架构结论，作为后续协议设计和代码实现的基线。

遵循 Superpowers 原则：

1. 先固定模块职责
2. 再固定标准任务模型
3. 再固定通信边界
4. 最后进入实现

## 1. 系统模块职责表

| 模块 | 核心职责 | 明确不负责 |
|---|---|---|
| `U8 / AI_MCU` | 人机交互入口；语音/按键/摄像头采集；与 `server` 建立会话；把上层任务转成对 `U1` 的设备级命令；回传状态和结果 | 不做运动控制闭环；不做底层实时插补；不持有复杂业务编排规则 |
| `Server` | 意图理解；任务生成；任务编排；会话与上下文管理；策略、权限、日志、任务持久化 | 不直接控制电机；不承载实时硬件时序；不依赖具体 GPIO/串口/Grbl 细节 |
| `U1 / MOTOR_MCU` | 执行运动/舵机/激光/传感器采集；归零、限位、安全停机；设备状态机；实时控制 | 不理解自然语言；不管理云端会话；不做跨任务编排 |

总原则：

- `Server` 负责任务“想什么”
- `U8` 负责任务“怎么转发”
- `U1` 负责任务“怎么安全执行”

## 2. 架构原则

### 2.1 Capability 原则

`server` 面向 capability 编程，不面向 controller 编程。

这意味着：

- `server` 知道机器有哪些标准能力
- `server` 不知道 `U1` 当前是 `Grbl` 还是其他控制器
- `U8` 是 `server` 与 `U1` 之间的唯一协议适配层
- `U1` 是最终执行层与安全裁决层

### 2.2 分层原则

按三层抽象组织：

- `L3` 语义任务层：用户目标、语音指令、AI 意图
- `L2` 设备无关能力层：`home`、`move_abs`、`run_path`
- `L1` 设备执行层：`HOME`、`MOVE`、`STATUS` 等串口/控制器命令

禁止跨层直连：

- `server` 不直接生成 `G0/G1/$H/?/M3` 这类控制器方言
- `U1` 不直接承担云端任务编排
- `U8` 不绕过 `server` 私自扩展业务规则

## 3. 标准任务模型

`server` 对外统一发标准任务，`U8` 负责翻译成 `U1` 可执行命令。

统一外层字段如下：

```json
{
  "task_id": "uuid",
  "task_type": "motion",
  "source": "voice|button|api",
  "target": "u1",
  "capability": "home|move_abs|run_path|get_status",
  "params": {},
  "constraints": {
    "requires_homing": true,
    "timeout_ms": 30000,
    "safety_level": "strict"
  }
}
```

## 4. 最小 Capability 集合

第一版只定以下 4 个 capability。

### 4.1 `home`

用途：建立已知坐标系。

```json
{
  "task_id": "t_home_001",
  "capability": "home",
  "params": {
    "axes": ["X", "Y", "Z"]
  },
  "constraints": {
    "timeout_ms": 60000,
    "safety_level": "strict"
  }
}
```

规则：

- 第一版默认只允许 `X/Y/Z`
- `Y2` 方正归零由 `U1` 内部决定
- 成功后 `homed=true`
- 失败后进入 `ALARM` 或 `ERROR`

### 4.2 `move_abs`

用途：移动到绝对坐标。

```json
{
  "task_id": "t_move_001",
  "capability": "move_abs",
  "params": {
    "x": 120.0,
    "y": 35.0,
    "z": 0.0,
    "feed_rate": 2000
  },
  "constraints": {
    "requires_homing": true,
    "timeout_ms": 30000,
    "safety_level": "strict"
  }
}
```

规则：

- 坐标单位统一为 `mm`
- 第一版只支持绝对坐标
- 未归零时 `U1` 可拒绝执行
- 超工作区时 `U1` 必须拒绝
- `feed_rate` 必须由 `U1` 做上限裁剪

### 4.3 `run_path`

用途：执行已规划好的路径，适用于画画、写字、描边。

```json
{
  "task_id": "t_path_001",
  "capability": "run_path",
  "params": {
    "coord_space": "canvas_mm",
    "pen_mode": "pen",
    "segments": [
      { "cmd": "M", "x": 10.0, "y": 10.0 },
      { "cmd": "L", "x": 30.0, "y": 10.0 },
      { "cmd": "L", "x": 30.0, "y": 30.0 }
    ],
    "feed_rate": 1200
  },
  "constraints": {
    "requires_homing": true,
    "timeout_ms": 120000,
    "safety_level": "strict"
  }
}
```

规则：

- 第一版只支持 `M` 和 `L`
- 曲线由上层先离散化
- 大路径由 `U8 -> U1` 分段传输
- `U1` 必须支持进度上报

### 4.4 `get_status`

用途：查询当前机器状态。

请求：

```json
{
  "task_id": "t_status_001",
  "capability": "get_status",
  "params": {}
}
```

返回建议：

```json
{
  "task_id": "t_status_001",
  "state": "IDLE",
  "homed": true,
  "position": {
    "x": 120.0,
    "y": 35.0,
    "z": 0.0
  },
  "active_task_id": null,
  "alarm_code": null,
  "error_code": null
}
```

## 5. 通信边界

### 5.1 `Server <-> U8`

只传三类东西：

- 会话事件：上线、心跳、鉴权、配置
- 交互事件：语音文本、按钮事件、图像摘要、上下文
- 任务事件：任务下发、任务状态、任务结果

这里使用“面向 capability/任务”的协议，不暴露 `U1` 的控制器细节。

### 5.2 `U8 <-> U1`

只传四类东西：

- 执行命令：`HOME`、`MOVE`、`PATH_BEGIN/SEG/END`、`STOP`、`GET_STATUS`
- 执行确认：`ACK`、`NACK`
- 状态上报：`IDLE`、`RUNNING`、`PAUSED`、`HOMING`、`ALARM`、`ERROR`、`ESTOP`
- 结果数据：位置、限位、传感器、错误码

这里使用“面向设备执行”的协议，不传自然语言语义。

## 6. `Server -> U8` 能力接口表

| capability | 说明 | 主要参数 | 返回 |
|---|---|---|---|
| `home` | 回零 | `axes` | `accepted/running/done/failed` |
| `move_abs` | 绝对移动 | `x,y,z,feed_rate` | `accepted/running/done/failed` |
| `run_path` | 执行路径 | `segments,feed_rate` | `accepted/running/done/failed` |
| `get_status` | 查询状态 | 无 | `state/homed/position/...` |

## 7. `U8 -> U1 (Grbl 适配)` 映射表

`U8` 可以使用私有串口协议与 `U1` 通信，`U1` 内部再适配为 `Grbl` 执行方式。

| capability | U8 发给 U1 的执行语义 | 若 U1 内部是 Grbl，可映射为 |
|---|---|---|
| `home` | `HOME` | `$H` |
| `move_abs` | `MOVE_ABS x y z f` | `G90` + `G0/G1 X..Y..Z..F..` |
| `run_path` | `PATH_BEGIN/SEG/END` | 分段展开为连续 `G1` |
| `get_status` | `GET_STATUS` | `?` |

关键原则：

- `server` 不直接生产 `G-code`
- `G-code` 只允许出现在 `U8/U1` 边界内部或 `U1` 内部
- 若 `U1` 后续替换为非 Grbl 控制器，`server` 无需改动

## 8. U1 状态集合

当前固定状态集合：

- `IDLE`
- `RUNNING`
- `PAUSED`
- `HOMING`
- `ALARM`
- `ERROR`
- `ESTOP`

## 9. U1 状态机转移表

当前只覆盖 `home`、`move_abs`、`run_path`、`get_status` 相关状态流转。

| 当前状态 | 触发 | 下一状态 | 说明 |
|---|---|---|---|
| `IDLE` | 接收 `home` 并通过校验 | `HOMING` | 进入归零流程 |
| `IDLE` | 接收 `move_abs` 并通过校验 | `RUNNING` | 进入点到点运动 |
| `IDLE` | 接收 `run_path` 并通过校验 | `RUNNING` | 进入路径执行 |
| `IDLE` | 接收 `get_status` | `IDLE` | 只读查询，不改状态 |
| `HOMING` | 归零完成 | `IDLE` | `homed=true` |
| `HOMING` | 归零失败 | `ALARM` 或 `ERROR` | 由失败原因决定 |
| `HOMING` | 接收 `estop` | `ESTOP` | 立即中断 |
| `RUNNING` | 任务完成 | `IDLE` | 返回完成态 |
| `RUNNING` | 接收 `pause` | `PAUSED` | 保持任务上下文 |
| `RUNNING` | 接收 `stop` | `IDLE` 或 `ALARM` | 正常停机成功则回 `IDLE`，异常则告警 |
| `RUNNING` | 接收 `estop` | `ESTOP` | 立即中断 |
| `RUNNING` | 运行中触发限位/超行程/严重异常 | `ALARM` 或 `ERROR` | 由故障类型决定 |
| `RUNNING` | 接收 `get_status` | `RUNNING` | 只读查询，不改状态 |
| `PAUSED` | 接收 `resume` 并通过校验 | `RUNNING` | 恢复执行 |
| `PAUSED` | 接收 `stop` | `IDLE` | 放弃当前任务 |
| `PAUSED` | 接收 `estop` | `ESTOP` | 立即中断 |
| `PAUSED` | 接收 `get_status` | `PAUSED` | 只读查询，不改状态 |
| `ALARM` | 接收 `get_status` | `ALARM` | 只读查询，不改状态 |
| `ALARM` | 故障清除并显式复位 | `IDLE` | 需要后续单独定义 `clear_alarm/reset` |
| `ERROR` | 接收 `get_status` | `ERROR` | 只读查询，不改状态 |
| `ERROR` | 故障清除并显式复位 | `IDLE` | 需要后续单独定义 `clear_error/reset` |
| `ESTOP` | 接收 `get_status` | `ESTOP` | 只读查询，不改状态 |
| `ESTOP` | 人工解除急停并显式复位 | `IDLE` | 必须强制人工参与 |

状态机约束：

- `ALARM`、`ERROR`、`ESTOP` 下禁止接受新的运动任务
- `get_status` 在任意状态都应可用
- `home`、`move_abs`、`run_path` 只能在允许状态下进入执行
- `ESTOP` 优先级最高，可从任何执行态切入

## 10. U1 错误码表

当前错误码只覆盖最小闭环。

| 错误码 | 名称 | 触发条件 | 建议状态 |
|---|---|---|---|
| `E001` | `NOT_HOMED` | 未归零却执行 `move_abs` 或 `run_path` | `ERROR` |
| `E002` | `OUT_OF_RANGE` | 目标坐标或路径超出工作区 | `ERROR` |
| `E003` | `INVALID_PARAM` | 参数缺失、类型错误、非法值 | `ERROR` |
| `E004` | `BUSY` | 当前状态不允许接受新任务 | `ERROR` |
| `E005` | `LIMIT_TRIGGERED` | 运行中触发限位 | `ALARM` |
| `E006` | `HOMING_FAILED` | 归零流程失败 | `ALARM` |
| `E007` | `MOTION_ABORTED` | 任务被 `stop` 主动终止 | `IDLE` 或 `ERROR` |
| `E008` | `ESTOP_TRIGGERED` | 急停触发 | `ESTOP` |
| `E009` | `SERIAL_PROTOCOL_ERROR` | 串口帧错误、校验失败、消息不完整 | `ERROR` |
| `E010` | `CONTROLLER_EXEC_ERROR` | 底层控制器执行失败，例如 Grbl 返回错误 | `ERROR` |
| `E011` | `TIMEOUT` | 任务执行超时 | `ERROR` |
| `E012` | `INTERNAL_STATE_ERROR` | 状态机内部不一致或不可恢复异常 | `ERROR` |

补充约定：

- `ALARM` 偏向机械/限位/归零类故障
- `ERROR` 偏向协议/参数/状态/执行类故障
- `ESTOP` 独立于一般错误码，表示最高优先级安全中断
- `MOTION_ABORTED` 是受控中止，不等同于机械故障

建议 `get_status` 返回时附带：

```json
{
  "state": "ERROR",
  "error_code": "E001",
  "alarm_code": null
}
```

## 11. U1 拒绝原则

以下情况 `U1` 必须能够拒绝执行：

- 未归零执行 `move_abs` 或 `run_path`
- 超行程
- 限位触发
- 当前状态不允许执行新运动命令
- 参数非法

安全裁决权最终属于 `U1`。

## 12. `U8 -> U1` 串口协议建议

第一版建议使用文本协议，便于抓串口日志和联调。

推荐帧形式：

```text
@{json}\n
```

示例：

```text
@{"msg_id":"101","type":"cmd","task_id":"t1","cmd":"HOME","axes":["X","Y","Z"]}
```

约束：

- 每帧以 `@` 开始
- 负载为单行 JSON
- 每帧以换行结束
- `msg_id` 由发送方生成，用于请求/响应关联
- `task_id` 用于任务级关联

## 13. `U8 -> U1` 串口消息字段表

### 13.1 公共字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `msg_id` | `string` | 是 | 消息唯一标识 |
| `type` | `string` | 是 | `cmd|ack|status|result|error` |
| `task_id` | `string` | 否 | 所属任务 ID；`get_status` 也建议携带 |

### 13.2 命令消息字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `cmd` | `string` | 是 | `HOME|MOVE|PATH_BEGIN|PATH_SEG|PATH_END|GET_STATUS|PAUSE|RESUME|STOP|ESTOP` |
| `axes` | `array[string]` | 条件必填 | `HOME` 使用 |
| `mode` | `string` | 条件必填 | `MOVE` 使用，第一版固定 `absolute` |
| `x` | `number` | 条件必填 | `MOVE` 或 `PATH_SEG` 使用 |
| `y` | `number` | 条件必填 | `MOVE` 或 `PATH_SEG` 使用 |
| `z` | `number` | 条件必填 | `MOVE` 使用 |
| `feed_rate` | `number` | 条件必填 | `MOVE` 或 `PATH_SEG` 使用，单位 `mm/min` |
| `segment_index` | `integer` | 条件必填 | `PATH_SEG` 序号，从 `0` 开始 |
| `segment_cmd` | `string` | 条件必填 | `PATH_SEG` 使用，第一版支持 `M|L` |
| `total_segments` | `integer` | 条件必填 | `PATH_BEGIN` 使用 |
| `coord_space` | `string` | 否 | `PATH_BEGIN` 使用，建议固定 `canvas_mm` |
| `pen_mode` | `string` | 否 | `PATH_BEGIN` 使用，建议固定 `pen` |

### 13.3 响应消息字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `state` | `string` | 条件必填 | `status` 使用，取值见状态集合 |
| `result` | `string` | 条件必填 | `result` 使用，例如 `DONE|ABORTED` |
| `error_code` | `string` | 条件必填 | `error` 使用 |
| `alarm_code` | `string` | 否 | `status/error` 使用 |
| `message` | `string` | 否 | 可读错误说明 |
| `homed` | `boolean` | 否 | 当前是否已归零 |
| `position` | `object` | 否 | 当前坐标 `{x,y,z}` |
| `active_task_id` | `string|null` | 否 | 当前活动任务 |
| `progress` | `number` | 否 | 路径执行进度，范围 `0.0 ~ 1.0` |
| `accepted` | `boolean` | 条件必填 | `ack` 使用，表示接受或拒绝 |

## 16. 实现约束

第一版实现时遵循以下约束：

- `U8` 必须维护 `task_id -> msg_id/segment sequence` 关联
- 大路径必须分段发送，不能一次性塞进单帧
- `PATH_BEGIN` 成功后才能发送 `PATH_SEG`
- `PATH_END` 只表示路径传输结束，不等于任务完成
- 任务真正完成以 `type=result` 且 `result=DONE` 为准
- `get_status` 在任何状态下都应可调用
- `ESTOP` 不依赖普通任务生命周期，必须最高优先级处理

## 17. 协议选型结论

第一版正式选择：

- `Server -> U8` 使用 capability 任务模型
- `U8 -> U1` 使用私有串口协议
- `U1` 内部再适配为 `Grbl` 或其他底层执行后端

明确不采用的方案：

- 不让 `server` 直接生成 `G-code`
- 不让 `U8 -> U1` 直接暴露 `G0/G1/$H/?/M3` 作为系统正式协议

选择私有协议的原因：

1. 便于携带 `task_id`、`msg_id`
2. 便于实现 `ACK/NACK`
3. 便于分段路径传输
4. 便于统一错误码和状态机
5. 便于后续替换 `U1` 内部控制器而不影响 `server`

一句话：

`Grbl` 仅作为 `U1` 内部执行后端，不作为系统对外协议。

## 18. U1 私有协议命令枚举

第一版命令集合如下：

| 命令 | 方向 | 说明 |
|---|---|---|
| `HOME` | `U8 -> U1` | 归零 |
| `MOVE` | `U8 -> U1` | 绝对移动 |
| `PATH_BEGIN` | `U8 -> U1` | 路径传输开始 |
| `PATH_SEG` | `U8 -> U1` | 单个路径段 |
| `PATH_END` | `U8 -> U1` | 路径传输结束 |
| `GET_STATUS` | `U8 -> U1` | 查询状态 |
| `PAUSE` | `U8 -> U1` | 暂停当前任务 |
| `RESUME` | `U8 -> U1` | 恢复当前任务 |
| `STOP` | `U8 -> U1` | 受控停止当前任务 |
| `ESTOP` | `U8 -> U1` | 急停 |
| `ACK` | `U1 -> U8` | 接收确认 |
| `STATUS` | `U1 -> U8` | 状态上报 |
| `RESULT` | `U1 -> U8` | 任务完成或受控结束 |
| `ERROR` | `U1 -> U8` | 错误/告警上报 |

补充约束：

- `PATH_SEG` 第一版仅支持 `M` 与 `L`
- `MOVE` 第一版仅支持 `absolute`
- `ESTOP` 必须可在任意状态打断
- `RESULT` 与 `ERROR` 必须携带 `task_id`

## 19. U8 Capability -> Serial Encoder 映射表

`U8` 适配层负责把 capability 请求编码为 `U1` 私有协议命令序列。

| capability | 输入 | U8 编码输出 |
|---|---|---|
| `home` | `axes` | 1 条 `HOME` |
| `move_abs` | `x,y,z,feed_rate` | 1 条 `MOVE` |
| `run_path` | `segments,feed_rate,...` | `PATH_BEGIN` + N 条 `PATH_SEG` + `PATH_END` |
| `get_status` | 无 | 1 条 `GET_STATUS` |

编码规则：

### 19.1 `home`

```json
{
  "task_id": "t_home_001",
  "capability": "home",
  "params": {
    "axes": ["X", "Y", "Z"]
  }
}
```

编码为：

```json
{"msg_id":"101","type":"cmd","task_id":"t_home_001","cmd":"HOME","axes":["X","Y","Z"]}
```

### 19.2 `move_abs`

```json
{
  "task_id": "t_move_001",
  "capability": "move_abs",
  "params": {
    "x": 120.0,
    "y": 35.0,
    "z": 0.0,
    "feed_rate": 2000
  }
}
```

编码为：

```json
{"msg_id":"102","type":"cmd","task_id":"t_move_001","cmd":"MOVE","mode":"absolute","x":120.0,"y":35.0,"z":0.0,"feed_rate":2000}
```

### 19.3 `run_path`

```json
{
  "task_id": "t_path_001",
  "capability": "run_path",
  "params": {
    "coord_space": "canvas_mm",
    "pen_mode": "pen",
    "segments": [
      { "cmd": "M", "x": 10.0, "y": 10.0 },
      { "cmd": "L", "x": 30.0, "y": 10.0 }
    ],
    "feed_rate": 1200
  }
}
```

编码为：

```json
{"msg_id":"103","type":"cmd","task_id":"t_path_001","cmd":"PATH_BEGIN","coord_space":"canvas_mm","pen_mode":"pen","total_segments":2,"feed_rate":1200}
{"msg_id":"104","type":"cmd","task_id":"t_path_001","cmd":"PATH_SEG","segment_index":0,"segment_cmd":"M","x":10.0,"y":10.0}
{"msg_id":"105","type":"cmd","task_id":"t_path_001","cmd":"PATH_SEG","segment_index":1,"segment_cmd":"L","x":30.0,"y":10.0,"feed_rate":1200}
{"msg_id":"106","type":"cmd","task_id":"t_path_001","cmd":"PATH_END"}
```

### 19.4 `get_status`

编码为：

```json
{"msg_id":"107","type":"cmd","task_id":"t_status_001","cmd":"GET_STATUS"}
```

## 20. 当前定稿结论

一句话总结：

`server` 不是通过“知道 U1 是 Grbl”来控制机器，而是通过调用标准 capability，由 `U8/U1` 适配层把 capability 映射为 Grbl 或其他底层执行方式。

### 22.2 当前未满足定稿的部分

#### U1 未满足项

私有协议骨架已存在但存在语义偏差，需要逐项收口：

- `report_private_status_json` 在 `State::Alarm` 时把 `alarm_code` 与 `error_code` 都硬编码为 `E006`（HOMING_FAILED）。按 §10 错误码表，限位触发应为 `E005`，归零失败才是 `E006`，ESTOP 应为 `E008`；而且 `alarm_code` 与 `error_code` 分别归属 ALARM/ERROR 两类，不应同值硬编码
- `homed` 字段当前直接由 `state != Alarm` 推断，未使用 Grbl 真实的归零完成状态，上电未归零且处于 IDLE 时会错报为 `true`，违反 §11 "未归零执行 move_abs/run_path 必须拒绝" 的语义基础
- 私有协议命令集缺口：尚未实现 `PAUSE / RESUME / STOP / ESTOP`，而 §9 状态机与 §18 命令枚举均要求这几条必须具备，`ESTOP` 应优先级最高
- `PATH_END` 采用同步执行：一次性把缓存的 G-code 跑完并 `protocol_buffer_synchronize`，运行期间无法产出 §13.3 要求的 `progress` 字段状态上报
- 尚未把 §10 错误码表的 `E001/E002/E003/E004/E007/E009/E011/E012` 等其他错误码映射到实际返回，现有实现只用到 `E003/E009/E010`
- `msg_id` 仅在请求回包里透传，尚未做服务端自发状态事件的 `msg_id` 规划

#### U8 未满足项

私有协议编码框架已具备，但存在明确的协议边界违规，需要按 §17 收口：

- `RunPath` 在 `PATH_BEGIN / PATH_SEG` 收到错误或超时时，回退到 `LoadJob(fallback_gcode) → StartJob` 的 G-code 直推路径。该回退违反 §17 "不让 U8 → U1 直接暴露 G0/G1/$H/?/M3 作为系统正式协议"，属于 §22.5 禁止的 "半协议半旧链路中间态"
- `self.motor.send_gcode` MCP 工具允许调用方注入任意 Grbl 命令行，等于把 Grbl 方言作为系统对外接口暴露，违反 §5.1 "Server → U8 不暴露 U1 控制器细节"
- `self.motor.jog` 自行拼出 `$J=G91 ... F...` 的 Grbl jog 语法直推，属于同类越界
- `self.motor.set_laser_power` 直接发 `M3/M5`，激光控制尚未收敛到 capability 模型
- `pause/resume` 使用 Grbl 实时控制字节 `!/~` 而非私有协议的 `PAUSE/RESUME`，等 U1 侧补齐这两条命令后应切回私有协议
- 尚未把 `U1` 返回的 `ack/status/result/error` 统一投影为 `accepted/running/done/failed` 的 server 可理解状态
- 尚未形成与 §3 标准任务模型对应的 `task_id` 生命周期管理，仅在单次请求内生成 `msg_id`

#### Server 未满足项

- 尚未收敛出 4 个最小 capability 的正式接口
- 尚未把高层写字/画图任务稳定投影为 `run_path`
- 尚未形成与 `U8` 对应的任务生命周期接口

### 22.3 当前链路的实际形态

当前实际形态是"新旧协议并存的中间态"，既不完全是旧 G-code 直推，也没有完全收敛到私有协议：

正在使用的新链路（capability 正路径）：

- `server` 或本地 MCP 工具下发 `capability`
- `U8` 将 capability 编码为 `@{json}\n` 私有协议帧
- `U1` 的 `execute_private_protocol_line` 解析并分发
- `U1` 内部把私有命令翻译为 `G0/G1/$H/G90` 下发给 Grbl 执行
- `U1` 通过 `grbl_sendf` 返回 `ack/status/result/error` JSON 行

并存的旧链路（需要收口的回退路径）：

- `U8` 侧 `RunPath` 在私有协议出错时回退到按行推送 G-code
- `U8` 侧 `send_gcode / jog / set_laser_power` 直接把 Grbl 方言透传给 `U1`
- `U8` 侧 `pause/resume` 使用 Grbl 实时控制字节 `!/~`，不走私有协议
- `U1/Grbl` 对这些行以传统的 `ok/error/ALARM` 风格响应

目标链路（§17 明确的收敛方向）：

- `server` 下发 `capability`
- `U8` 只编码为私有协议，不再保留 G-code 回退
- `U1` 解析私有协议
- `U1` 内部适配到 `Grbl`
- `U1` 统一返回 `ACK/STATUS/RESULT/ERROR`，不再暴露 `ok/error/ALARM` 原文给上层

### 22.4 闭环实施顺序

严格按最小闭环推进，不同时改所有层。每条都标注当前状态，避免重复劳动：

1. 在 `U1` 加 `GET_STATUS` 私有协议入口 —— 骨架已完成；待修：`homed` 字段需对接 Grbl 真实归零状态、`alarm_code/error_code` 需按 §10 区分
2. 在 `U8` 加 `GET_STATUS` 编码与响应解析 —— 编码侧已完成；待补：`U1` 返回投影为 `accepted/running/done/failed`
3. 打通 `HOME` —— 骨架已完成；待修：返回的 `result/state` 需基于真实归零完成事件而非 `execute_line` 立即返回值
4. 打通 `MOVE` —— 骨架已完成；待修：未归零拒绝逻辑应返回 `E001 NOT_HOMED`，超行程返回 `E002 OUT_OF_RANGE`
5. 打通 `run_path` 的 `PATH_BEGIN/PATH_SEG/PATH_END` —— 同步执行骨架已完成；待补：异步执行 + `progress` 进度上报；待删：U8 侧 G-code 回退路径
6. 补齐控制流：`PAUSE/RESUME/STOP/ESTOP` 私有协议入口 —— 未开始
7. 收口越界 MCP 工具：`send_gcode / jog / set_laser_power` 改为 capability 形式或下沉到 U1 内部 —— 未开始
8. 把现有 `G-code job` 流程整体移除或仅保留为 U1 内部调试态 —— 未开始
