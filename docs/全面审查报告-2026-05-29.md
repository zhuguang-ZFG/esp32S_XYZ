## 一、项目概况

双 ESP32-S3 板级集成项目，面向儿童/家庭的 AI 创意教育产品（写字机）：

| 组件 | 技术栈 | 作用 |
|------|--------|------|
| U1 (MOTOR_MCU) | Grbl_Esp32, PlatformIO, C/C++ | CNC 运动控制（4轴 XYYZ 步进电机） |
| U8 (AI_MCU) | xiaozhi-esp32, ESP-IDF, C++ | 语音唤醒、ASR/LLM/TTS、摄像头、显示 |
| DeviceServer | Python, WebSocket | 设备会话管理、ASR/LLM/TTS 调度 |
| BusinessServer | Java 21, Spring Boot | 账号、设备绑定、任务生命周期、内容审核 |
| Mobile Client | uni-app, 微信小程序 | 设备控制、任务提交、实时状态 |

**里程碑状态**：M0-M6 设计阶段静态验证已通过，243+ CI 测试，76 Java 测试，62 Schema 验证。

## 二、痛点分析（按严重度排序）

### P0 — 必须立即修复

#### 1. U1 缓冲区溢出（Report.cpp:606）

`report_realtime_status()` 用 `char status[200]` + 30+ 次 `strcat()` 拼接状态字符串。所有字段启用时接近或超过 200 字节。SD 文件名追加无长度检查。

**风险**：堆栈溢出导致固件崩溃/重启。

#### 2. switch fallthrough bug（Report.cpp:728-738, 751-761）

```cpp
case State::SafetyDoor:
    sys.report_wco_counter = (REPORT_WCO_REFRESH_BUSY_COUNT - 1);
default:  // ← 直接落入，覆盖为 IDLE 值
    sys.report_wco_counter = (REPORT_WCO_REFRESH_IDLE_COUNT - 1);
```

**影响**：忙碌状态下始终使用空闲刷新率，导致状态上报延迟。

#### 3. WebServer 认证缓冲区溢出（WebServer.cpp:631-632）

`strcpy(current_auth->sessionID, ...)` 和 `strcpy(current_auth->userID, ...)` 无边界检查。HTTP 请求中的用户名可远程触发溢出。

**风险**：远程可利用。

#### 4. WebServer 内存泄漏（WebServer.cpp:628）

`new AuthenticationIP` 在 `AddAuthIP()` 返回 false 时未释放。

### P1 — 尽快修复

#### 5. U1 JSON 解析器脆弱（Protocol.cpp:50-103）

用 `strstr()` 搜索 `"field":` 模式。字段顺序变化、引号嵌套、空白变化都会解析失败。这是运行时最高风险。

#### 6. 单一错误码 E009（Protocol.cpp 14 处）

所有协议错误（home 失败、缺少轴目标、移动失败、暂停无效等）都返回 E009，客户端无法区分错误类型。

#### 7. U8 板文件是单体（dlc_motor_control_p1_ai_board.cc, 1089 行）

音频编解码、I2C/Camera/UART 初始化、MCP 工具注册、U8-U1 协议层、运动任务分发、WebSocket 全部在一个文件中。协议层无法复用。

#### 8. 无固件 CI

U1 和 U8 固件都不在 CI 中编译。C/C++ 代码回归完全不可见。

#### 9. API Key 残留在 git 历史中

`tools/test_vectorize.py` 已修复为 `os.environ.get()`，但旧 key 仍在审计文档和 git 历史中。

### P2 — 改进项

#### 10. 无构建/烧录文档

新开发者无法知道：安装什么工具、选什么板配置、怎么编译烧录、怎么跑测试。

#### 11. U8 CMakeLists.txt 700 行 if/else 链

板选择应数据驱动，不是巨型条件分支（`main/CMakeLists.txt` 92-816 行）。

#### 12. 固件 C/C++ 单元测试为 0

关键嵌入式代码（JSON 解析、协议处理、状态上报）无自动化测试。

#### 13. 无顶层构建脚本

无 Makefile / build.sh / justfile。两个 MCU 用不同工具链，无统一入口。

#### 14. U8 cJSON 使用不一致

解析用 cJSON，构造用原始字符串拼接（`BuildProtocolCommandJson`），存在注入风险。

#### 15. `ReturnValue` variant 类型

`std::variant<std::string, cJSON*>` 需要每个调用点手动检查类型和释放内存，泄漏风险高。

### P3 — 技术债

#### 16. 死代码

- U1: 27 个无关 machine 头文件、4 处 `#if 0` 块
- U8: ~100+ 上游 board 目录（仅 1 个是自定义的）
- tools: `test_vectorize.py` 不在 CI 中

#### 17. 依赖范围过宽

`platformio.ini` 中 `TMCStepper@>=0.7.0,<1.0.0` 未收紧。Python 工具无 requirements.txt。

#### 18. 重复实现

- `fake_u1` 有两个竞争实现（async vs sync）
- `report_hex_msg` 有两个几乎相同的重载（Report.cpp:914-936）
