# M6.7 Device RMA Status

Status: implemented for manager-api minimal backend flow.

Evidence Boundary:

This status is local/design-time evidence only. It is not a completion declaration and does not replace production operator permission assignment, gateway policy evidence, exported audit review, factory credential cleaning proof, or deployed DeviceServer refusal evidence.

Implemented:

- Added device RMA service endpoints:
  - `POST /api/v1/devices/{deviceId}/rma/start`
  - `POST /api/v1/devices/{deviceId}/rma/complete`
  - `POST /api/v1/devices/{deviceId}/return`
  - `POST /api/v1/devices/{deviceId}/restock`
  - `POST /api/v1/devices/{deviceId}/dispose`
  - `POST /api/v1/devices/{deviceId}/rma/events`
- RMA endpoints allow either the existing manager-api `sys:role:superAdmin` permission or the narrower `appv2:device:rma` permission, so repair, return, restock, and dispose actions can be operator/admin gated without granting full super-admin access.
- RMA transitions write local `device_rma_events` audit rows with operator account id, device id/sn, action, previous/new device status, previous/new binding status, optional `ticketRef`, and timestamp. The audit row intentionally does not store activation codes or free-form RMA notes.
- Restock and dispose require local factory cleaning attestation: `factoryCleaned = true` and a non-empty `evidenceRef`. The evidence reference is stored on the RMA audit row; activation codes and free-form notes are not stored there.
- RMA operators can list the latest local audit rows through the local audit export endpoint `POST /api/v1/devices/{deviceId}/rma/events` for evidence export and external ticket correlation.
- Repair flow:
  - `bound -> rma_in_progress`
  - active binding is selected by `updated_at` desc and binding `id` desc, then becomes `rma_in_progress`, preserving `account_id`
  - `rma_in_progress -> bound`
  - latest RMA-in-progress binding is selected by the same order and returns to `active`
- Return flow:
  - `bound -> returned`
  - active binding is selected by the same deterministic order, then becomes `returned`, preserving account-side history linkage
  - activation code is revoked
  - `returned -> provisioned_unbound`
  - activation code is refreshed and marked `provisioned` for resale, using an approved replacement code when supplied or generating a fresh code when omitted
  - activation-code updates select the latest row for the device by `updated_at` desc and activation `id` desc before revoking or refreshing it
- Dispose flow:
  - any known device state can be marked `disposed`
  - the latest known binding is selected deterministically by `updated_at` and `id`, then becomes `disposed`
  - activation code is revoked
  - BusinessServer internal runtime entrypoints reject disposed devices with `E_DEVICE_DISPOSED`
  - DeviceServer voiceprint cache refresh clears any stale local cache when BusinessServer returns `E_DEVICE_DISPOSED` or `DEVICE_NOT_EXIST`
  - DeviceServer voice-task submission treats BusinessServer `Result.code != 0` as a failed submission, so a disposed-device refusal is not announced as accepted
  - DeviceServer `motion_event`, `device_info`, and `self_check` forwarders treat BusinessServer `Result.code != 0` as rejected forwarding, so disposed-device refusals are not logged as successful uploads
  - xiaozhi-server queries `POST /config/device-runtime-status` during WebSocket auth and rejects explicitly disposed devices with `E_DEVICE_DISPOSED` before local whitelist/token bypass can admit the socket
  - when Manager API runtime-status checking is configured but unavailable, xiaozhi-server rejects WebSocket auth with `E_RUNTIME_STATUS_UNAVAILABLE` instead of falling back to the local whitelist/token path
- Added migration widening `devices.status`, `device_bindings.binding_status`, and `activation_codes.status` to fit v2 RMA states.
- Added migration `202605160220.sql` for the local `device_rma_events` operator audit table.
- Added operations runbook `ops/runbooks/m6-device-rma.md` covering operator access requirements, factory credential cleaning, disposed-device refusal drills, restock drills, and evidence attachment.

Limitations:

- This slice exposes a granular `appv2:device:rma` manager-api permission alongside `sys:role:superAdmin`, requires local factory-cleaning attestation for restock/dispose, records local structured RMA audit rows, and supports local audit export with `ticketRef`; production role assignment, gateway policy, exported audit evidence review, and physical factory cleaning proof are still future hardening.
- xiaozhi-server WebSocket auth refusal depends on configured `manager-api.url` and `manager-api.secret`; local-only DeviceServer deployments without manager-api configured still rely on BusinessServer internal runtime refusal, while configured-but-unreachable Manager API checks fail closed.
- Factory credential cleaning remains an offline production process; the runbook defines required evidence, but this backend slice only resets BusinessServer status and activation-code state.
