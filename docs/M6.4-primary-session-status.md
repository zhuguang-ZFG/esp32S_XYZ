# M6.4 Primary Session Status

Date: 2026-05-16

## Evidence Boundary

This status is local/design-time evidence only. It is not a completion declaration and does not replace platform push delivery, native tabBar screenshots on real WeChat clients, background reminder delivery, or approved release deferral evidence.

## Scope

- Manager API enforces a single-writer rule: client write tasks must come from the currently claimed primary session for the bound account/device.
- Edge-A client WebSocket supports `claim_primary`, so an authenticated session can claim primary control.
- Voice writes respect primary priority: when an unexpired primary session exists, voice write tasks are stored as `pending_primary_approval` and are not forwarded to DeviceServer until the primary explicitly approves them.
- The primary can approve or reject each pending voice task individually.
- The mobile device detail page can list pending voice approvals for the current bound device and approve/reject them from the primary client.
- Non-primary client writes return `E_NOT_PRIMARY`; subscription and replay remain read-only.
- Claimed primary sessions have a 60 second lease. Expired leases are released before client write checks and before voice-write gating.

## Implementation

- `PrimarySessionService` owns primary write validation, voice-write gating, and primary claims.
- `PrimarySessionServiceImpl` validates active device binding ownership and uses:
  - `accounts.primary_session_id` as the current primary session id.
  - `accounts.primary_session_claimed_at` as the lease timestamp.
  - `PRIMARY_SESSION_LEASE_MS = 60_000L` as the lease window.
- `AppV2ServiceImpl.submitTask` calls `requirePrimaryForWrite(accountId, deviceId, SecurityUser.getToken())` before resource deduction, task persistence, and forwarding.
- `AppV2ServiceImpl.submitVoiceTask` calls `requireVoiceAllowedForWrite(accountId, deviceId)`. If a primary session blocks the voice write, it persists a `pending_primary_approval` task and returns `approvalRequiredBy = primary` without resource deduction or DeviceServer forwarding.
- Voice-originated tasks resolve the active binding deterministically before primary-session arbitration by selecting the latest active binding ordered by `updated_at` desc, then binding `id` desc.
- client write, primary claim, pending-approval listing, and Edge-A device subscription checks also select account/device active bindings by `updated_at` desc, then binding `id` desc, so duplicate anomalous active rows do not depend on database scan order.
- `AppV2ServiceImpl.approveVoiceTask` and `rejectVoiceTask` require the current primary session before deciding a specific pending voice task:
  - approve re-runs write safety/resource checks, marks the task `accepted`, and forwards it to DeviceServer.
  - reject marks the task `rejected` with `E_PRIMARY_REJECTED` and never forwards it.
- REST endpoints:
  - `POST /api/v1/tasks/{taskId}/approve`
  - `POST /api/v1/tasks/{taskId}/reject`
  - `POST /api/v1/devices/{deviceId}/voice-tasks/pending`
- Manager mobile exposes pull-based primary review on `pages/v2/device-detail/index.vue`:
  - `v2ListPendingVoiceTasks(deviceId)` loads pending `voice` tasks.
  - `v2ApproveVoiceTask(taskId)` approves and forwards the selected task.
  - `v2RejectVoiceTask(taskId)` rejects the selected task without forwarding.
  - The page shows an in-app pending approval count badge and a refreshable empty state.
  - The native home tabBar badge is refreshed after pending voice-approval pulls and reapplied from local pending counts when the app returns to `onShow`.
- Product notification runbook `ops/runbooks/m6-product-notifications.md` defines the future platform push/background reminder contract and acceptance drill for pending primary voice approvals.
- `ProductNotificationOutboxService` writes a `pending_primary_voice_approval` row to `product_notification_events` when a voice task is held for primary approval. The row stores only safe semantic fields and a deep link, not prompt text, transcript text, or biometric material.
- The local outbox enqueue API for primary voice approvals receives only recipient, device, and task references; it does not accept capability, prompt, transcript, or other task payload fields.
- `ProductNotificationOutboxService.buildSafeProviderPayload(...)` exposes only `event`, `device_id`, `target_ref_type`, `target_ref_id`, and `deep_link` for a future provider worker. It does not expose recipient account id, outbox status, capability, prompt, transcript, biometric material, or activation codes.
- `ProductNotificationOutboxService.markProviderSent(...)` and `markProviderFailed(...)` update only still-pending outbox rows by id, so a future provider worker can feed `sent` and `failed` lifecycle state without mutating resolved/cancelled rows.
- Pending primary voice approval outbox rows are marked `resolved` when the primary approves the task and `cancelled` when the primary rejects it, so a future push worker does not keep scanning stale pending rows.
- `ClientEdgeWebSocketHandler` stores the authenticated token as the session id and supports:
  - `{"op":"claim_primary","device_id":"..."}`
  - success response `{"type":"primary_claimed","device_id":"...","role":"primary"}`
- Edge-A client WebSocket auth rejects `status = deleted` account tombstones. Existing sessions revalidate the stored token and account status before every non-`auth` message, so account deletion or token logout closes stale sessions before subscribe or primary-claim handling.
- `RenExceptionHandler` maps `PrimarySessionException` to `FORBIDDEN`, with response text containing `E_NOT_PRIMARY`.
- Device transfer and account deletion clear both `primary_session_id` and `primary_session_claimed_at`.

## Validation

- `PrimarySessionServiceImplTest` covers:
  - matching primary session allows client writes.
  - non-primary client session returns `E_NOT_PRIMARY`.
  - unclaimed primary allows voice writes.
  - unexpired primary blocks voice writes with `E_NOT_PRIMARY`.
  - `claim_primary` stores the current session and lease timestamp.
  - expired primary lease is released before client write validation.
  - expired primary lease is released before voice-write gating.
- `AppV2ServiceImplTest` covers:
  - non-primary client writes are rejected before persistence and forwarding.
  - voice writes become `pending_primary_approval` and are not forwarded when a primary session blocks them.
  - primary approval forwards a specific pending voice task.
  - primary rejection marks a specific pending voice task rejected and does not forward it.
- `AppV2ControllerTest` covers approve/reject/list-pending endpoint delegation.
- `ClientEdgeWebSocketHandlerTest` covers `claim_primary` delegation and acknowledgement.
- `ClientEdgeWebSocketHandlerTest` covers deleted account auth refusal and token revalidation before a previously authenticated session can subscribe to a device.

## Limitations

- The mobile surface is pull-based from the device detail page. It updates a best-effort native tabBar badge after pending voice-approval refreshes and reapplies the stored local badge count on app `onShow`; a local pending notification outbox plus safe provider payload builder exist for future push workers. Platform push notifications, native tabBar rendering on every platform, and background reminders for pending voice approvals are not proven locally. If push or reminders are deferred for release, the release ticket must record the deferral owner, due date, fallback path, and risk acceptance.
