# M5.4 OTA client status

## Scope

This slice tightens the existing U8 OTA client path into a design-time OTA contract for A/B firmware update.

## Implemented

- The existing partition table already keeps `ota_0` and `ota_1` A/B app slots.
- `sdkconfig.defaults` keeps `CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE=y`.
- `Ota::CheckVersion()` now treats release metadata as valid only when a new firmware entry includes:
  - `url` with `https://`
  - `sha256` as 64 lowercase hex chars
  - `signature` as a base64 field
- `Ota::Upgrade()` refuses non-HTTPS firmware URLs.
- `Ota::Upgrade(url, expected_sha256, callback)` streams the downloaded image through mbedTLS SHA-256 and compares it before `esp_ota_end()` / `esp_ota_set_boot_partition()`.
- `Ota::StartUpgrade()` passes the release metadata SHA-256 and signature into the upgrade path.
- `Application::UpgradeFirmware(...)` now preserves SHA-256 and signature metadata from the OTA check response instead of falling back to the URL-only debug path.
- Metadata-based upgrades verify the base64 release signature over the downloaded image SHA-256 digest with `CONFIG_OTA_VERIFY_PUBLIC_KEY_PEM` before `esp_ota_end()` / `esp_ota_set_boot_partition()`.
- `Application::UpgradeFirmware()` rejects OTA when the device is not `idle` or `activating`, so listening/speaking/task states do not enter upgrade.
- U8 parses `firmware.release_id`, stores a pending OTA install result before reboot, reports failure when local upgrade fails, and reports success after `esp_ota_mark_app_valid_cancel_rollback()`.
- Static CI covers A/B rollback config, metadata fields, HTTPS guard, SHA-256 verification, signature verification, and device-state gate.

## Not yet complete

- Real ESP-IDF U8 build after sdkconfig resolution.
- Real HTTPS certificate-chain validation against production OTA host.
- Provisioning a production OTA public key and signing private-key custody. The client-side verification path exists, but `CONFIG_OTA_VERIFY_PUBLIC_KEY_PEM` must be set by the release build.
- End-to-end OTA from `ota_0` to `ota_1`, reboot, reconnect, and `esp_ota_mark_app_valid_cancel_rollback()` confirmation on hardware.
- Hardware evidence that the post-reboot install-result POST reaches DeviceServer / BusinessServer.
- Backend release metadata service and gray rollout logic; those belong to M5.5.

## OTA Client Evidence Gap Record

Use this structure when OTA client evidence is missing for a release candidate. It is not evidence of a real U8 A/B OTA rollout.

- missing evidence scope: resolved ESP-IDF U8 build, production HTTPS certificate-chain validation, OTA public-key provisioning, signing private-key custody, hardware A/B OTA from `ota_0` to `ota_1`, rollback validation, `esp_ota_mark_app_valid_cancel_rollback()` confirmation, or post-reboot install-result delivery
- environment: staging U8 device, release-candidate firmware, ESP-IDF version, partition table, resolved sdkconfig, OTA host, and release metadata record
- owner: named firmware or security owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep OTA labeled design-time, keep manual recovery or serial flashing available, and block production rollout if signature verification, rollback, or install-result delivery is unproven
- risk acceptance: firmware owner, security owner, operations owner, and release manager approval reference
- rollback trigger: ESP-IDF build fails, certificate chain is invalid, public key is missing, signature verification fails, A/B slot switch fails, rollback does not restore the previous app, or install-result delivery is not observed
- follow-up evidence: build artifact id, resolved sdkconfig, redacted public-key provisioning record, signing custody approval, OTA metadata, certificate-chain validation output, serial OTA log, `ota_0` to `ota_1` partition evidence, rollback drill log, and DeviceServer / BusinessServer install-result receipt

## Current security posture

This slice prevents accidental plain HTTP update URLs and corrupted image installs through SHA-256 mismatch detection. Metadata-based upgrades now also require a base64 signature and verify it with the configured OTA public key before boot partition switching. The client-side install-result reporting path exists, but production key provisioning and hardware OTA evidence remain open.
