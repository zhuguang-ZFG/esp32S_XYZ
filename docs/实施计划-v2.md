# 实施计划-v2.md

把 `架构定稿-v2.md §17 联调顺序` 的十步展开为按文件、按函数的具体计划。每一步包含：

- 前置条件
- 具体修改的文件 / 函数 / 行
- 完成判定
- 风险与回退

本计划保持与 v2 §16 实施清单一致。任何冲突以 v2 为准。

## 0. 总图

```
里程碑 M0  设计期验证（无主板可做）            ← 本地闭环已完成，M0f 等实物
   M0a 契约固化（JSON Schema）
   M0b 静态检查（GPIO 冲突、strapping pin）
   M0c 仿真器（fake U1、fake DeviceServer、fake provider）
   M0d CI 与单元测试骨架
   M0e 基准修复（已完成）
   M0f 实物上电抽测                        ← 标"等实物"，不阻塞 M1~M3 软件设计
里程碑 M1  设备协议端到端 (GET_STATUS/HOME/MOVE) ← 当前软件轨推进
里程碑 M2  小程序订阅链路打通 (run_path 可见)
里程碑 M3  write_text/draw 投影端到端
里程碑 M4  语音触发 + 声纹
里程碑 M5  OTA / 配网 / 自检
里程碑 M6  上线合规与运营级功能
```

设计期纪律：

- 没有主板时，**所有能在没主板下做的验证一律前移到 M0**
- 主板到货前 M1~M3 全部做"对着 fake 设备"的开发与单测
- 主板到货后用最少时间走 M0f + 真实电机加速度调参，立刻进入真实联调
- 这条纪律出自 Superpowers 的"先验证后实现"精神：尽早把不确定性收敛到可被自动化抓住的形态

每个 M 必须按顺序，每个 M 结束后跑回归再开下一个。

## 文档同步规则

任何 M 的实施过程中如果发现 v2 定稿与现实不符，**先停下，更新 v2，再继续**。

修订流程：

1. 在 `docs/架构定稿-v2.md` 对应小节修订
2. 修订内容附"修订记录"行（日期 + 修订原因）
3. 关联本计划相应 M.x 也更新
4. 才能继续动代码

绝不允许"代码先改了，文档后面再说"——基准必须先对齐。
