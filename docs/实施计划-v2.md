# 实施计划-v2.md

把 `架构定稿-v2.md §17 联调顺序` 的十步展开为按文件、按函数的具体计划。每一步包含：

- 前置条件
- 具体修改的文件 / 函数 / 行
- 完成判定
- 风险与回退

本计划保持与 v2 §16 实施清单一致。任何冲突以 v2 为准。

## 0. 总图

```
里程碑 M0  基准修复（已部分完成）         ← 现在所在位置
里程碑 M1  设备协议端到端 (GET_STATUS/HOME/MOVE)
里程碑 M2  小程序订阅链路打通 (run_path 可见)
里程碑 M3  write_text/draw 投影端到端

里程碑 M5  OTA / 配网 / 自检
里程碑 M6  上线合规与运营级功能
```

每个 M 内有 1~3 个 v2 §17 步骤。M 必须按顺序，每个 M 结束后跑回归再开下一个。

### M0.4 实机抽测 U1 弱证 GPIO（待办，**进入 M1 前**）

弱证 21 条 U1 GPIO 必须实测才能视为基准。最小验证集合：

- 上电不烧 Grbl，仅烧"GPIO blink test"固件，依次让以下脚翻转 1Hz：
  IO3 / IO4 / IO5 / IO6 / IO8 / IO16 / IO17 / IO18 / IO46 — 应观察对应步进 IC 的 STEP/DIR/EN 翻转
  IO45 — 应观察激光 MOSFET 翻转
- 然后让以下脚做输入并打印电平：
  IO9 / IO12 / IO13 / IO14 — 用手指或铁片触发原点传感器，应见电平变化
- 舵机 IO21 / IO47 / IO48、HX711 IO7 / IO15 在不需要时不在本最小集

**完成标准**：每条信号点动后能在对应外设上观察到现象。任一不通即按表回查 EDA。

风险：blink test 期间 STEP 翻转会让电机走，**测试时要拆下机械连接或松螺丝**。

## M1 设备协议端到端（v2 §17.1~§17.2）

目标：从 U1 直接发 `GET_STATUS / HOME / MOVE` 私有协议帧，能拿到正确响应。

### M1.1 修 U1 `Report.cpp::report_private_status_json` 的 `homed` 字段

- 文件：`firmware/u1-grbl/Grbl_Esp32/src/Report.cpp:829-854`
- 现状：`homed` 字段直接由 `state != Alarm` 推断
- 改为：读 Grbl 的 `sys.homed_axes` 真实归零位
- 完成判定：未归零开机时 `GET_STATUS` 返回 `"homed":false`；归零成功后返回 `"homed":true`
- 风险：若 Grbl 暴露的 `sys.homed_axes` 字段命名不同，需查 Grbl_Esp32 v0.8 实际 API
- 回退：保留旧推断作为默认分支，仅在确认 API 存在时启用

### M1.2 修 `alarm_code` / `error_code` 区分

- 文件：同上 Report.cpp
- 现状：ALARM 时 `alarm_code` 与 `error_code` 都硬编码为 `E006`
- 改为：按 v1 §10 错误码表，限位触发 → `E005`，归零失败 → `E006`，急停 → `E008`
- 数据来源：`sys.state` + `sys_rt_exec_alarm` 枚举值
- 完成判定：触发限位时事件含 `alarm_code:E005` 而非 E006

### M1.3 U1 实现 `PAUSE / RESUME / STOP / ESTOP` 私有命令

- 文件：`firmware/u1-grbl/Grbl_Esp32/src/Protocol.cpp::execute_private_protocol_line` (line 130-)
- 现状：仅识别 GET_STATUS / HOME / MOVE / PATH_BEGIN / PATH_SEG / PATH_END
- 改为：补 PAUSE → 发 Grbl `!`（feed hold）；RESUME → `~`；STOP → `|`（如 Grbl 0.8 不支持则用 `\x18`）；ESTOP → 立即调用现有 ESTOP 处理函数
- 完成判定：每条命令都返回 `type:"ack"`，行为生效
- 风险：U1 BOOT 短按软急停（v2 §10bis.12）需要单独中断处理，不在本步内

### M1.4 U1 `at_home` 派生位字段补全

- 文件：Report.cpp + Settings.cpp
- 现状：U1 不计算 `at_home`，让 BusinessServer 派生
- 改为：U1 在 `report_private_status_json` 多输出 `home_position` 与 `position_mm`，让 BusinessServer 能比对
- v2 约定 BusinessServer 派生 `at_home`，U1 不计算

### M1.5 删除 U8 G-code 回退路径（v2 §16.4 第 2 项）

- 文件：`firmware/u8-xiaozhi/main/boards/zhuguang/dlc-motor-control-p1-ai/dlc_motor_control_p1_ai_board.cc::RunPath` (line 534-626)
- 现状：私有协议失败时回退 `LoadJob(fallback_gcode) + StartJob`
- 改为：完全删除 fallback 分支，错误直接返回错误码
- 关联清单：`self.motor.send_gcode` MCP 工具同时移除（v2 §16.4 第 3 项）；`self.motor.jog` 改为内部用私有 MOVE 协议而非 `$J=G91` 字串
- 完成判定：grep 全文不再出现 `LoadJob` / `$J=G91` / `M3 S` 这类 Grbl 方言字串（除 U1 内部）

### M1.6 U8 私有协议直发 GET_STATUS / HOME / MOVE 通过

- 端到端测试：从 U8 串口控制台或 MCP 工具发起，U1 应分别响应
- 完成判定：三条 capability 各跑通 5 次无误

- 完成判定：DeviceServer 端 log 显示已派发；U8 端 log 显示已收到

### M2.5 U8 接 motion_task → 编码为私有协议发 U1

## 文档同步规则

任何 M 的实施过程中如果发现 v2 定稿与现实不符，**先停下，更新 v2，再继续**。

修订流程：

1. 在 `docs/架构定稿-v2.md` 对应小节修订
2. 修订内容附"修订记录"行（日期 + 修订原因）
3. 关联本计划相应 M.x 也更新
4. 才能继续动代码

绝不允许"代码先改了，文档后面再说"——基准必须先对齐。

## 风险登记

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| U1 弱证 GPIO 实测发现错位 | 中 | 高，需重排电路 | M0.4 一定要在 M1 前完成 |

| OTA 灰度遇大面积失败 | 中 | 极高 | 灰度上限 10% 起，监控见错就停 |
