# M5.8 小程序健康检查状态

日期：2026-05-15

## 已完成

- 在小程序 V2 设备详情页增加“健康检查”区域。
- 用户可手动点击健康检查按钮，提交 `run_path` 任务。
- 健康检查路径为保守方框空行程：
  - `(5,5,0) -> (25,5,0) -> (25,25,0) -> (5,25,0) -> (5,5,0)`
  - `feed: 900`
- 页面接入 Edge-A `self_check` 事件，展示最近一次诊断状态、时间和摘要。
- 摘要覆盖 M5.7 自检项：`nvs`、`wifi`、`u1_uart`、`audio`。
- 健康检查提交和任务终态复用既有 `job_status` 进度链路。

## 验收口径

- 小程序端存在健康检查按钮。
- 按钮触发 `v2SubmitTask(deviceId.value, 'run_path', ...)`。
- 页面收到 `event_type: "self_check"` 后更新最近一次诊断摘要。
- 静态 CI 覆盖健康检查 UI、`run_path` 提交和 `self_check` 摘要关键 token。

## 当前限制

- 方框空行程是否真实安全执行，仍需连接真实 U1/机械结构后验证。
- 小程序展示的是最近一次上报摘要，不包含诊断历史持久化。
- M5.8 复用 M5.7 的 `self_check` 上报链路；深度诊断的更多传感器闭环可在后续里程碑扩展。
## 2026-05-16 Local History Update

- Backend now persists `self_check` events in `device_self_check_events`.
- App V2 exposes `POST /api/v1/devices/{deviceId}/self-check/history` for the latest five diagnostics of a bound device.
- Mini program device detail loads and displays recent self-check history, and prepends live Edge-A `self_check` events into the history list.
- Static CI covers the table, DAO/entity/DTO, ingest persistence, history query endpoint, mobile API, and recent-history UI tokens.
- This supersedes the earlier local limitation text that said diagnostic history persistence was not included.

Remaining limits:

- Real U1/mechanism safety for the square health-check path still needs hardware verification.
- Historical rows are persisted locally, but no production retention policy or dashboard has been deployed for this table yet.

## Health Check UI Evidence Gap Record

Use this structure when health-check UI evidence is missing for a release candidate. It is not evidence of real U1/mechanism safety or production diagnostic retention.

- missing evidence scope: real U1/mechanism safety for the square health-check path, mobile health-check button drill, `run_path` submission, `self_check` event rendering, diagnostic history retrieval, production retention policy, or dashboard visibility for `device_self_check_events`
- environment: staging mini program build, physical phone model, WeChat version, bound U8/U1 device, release-candidate firmware, BusinessServer deployment, and Edge-A event channel
- owner: named mobile, firmware, backend, or operations owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep health-check UI labeled local/static, keep the square path conservative, hide or disable production-facing health-check claims if hardware safety or history retention is unproven, and block release if unsafe movement is observed
- risk acceptance: mobile owner, firmware owner, backend owner, operations owner, and release manager approval reference
- rollback trigger: square health-check movement is unsafe, `run_path` submission fails, live `self_check` event does not render, history endpoint omits recent diagnostics, retention policy is missing for production, or dashboard visibility is required but unavailable
- follow-up evidence: phone model, WeChat version, mini program build id, square-path hardware video or serial log, `run_path` request/response, Edge-A `self_check` event screenshot, history endpoint response, retention approval, and dashboard screenshot or explicit dashboard deferral
