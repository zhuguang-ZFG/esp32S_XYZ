## 1. 商业定位与约束

### 1.1 商业主线

主线：儿童/家庭创意教育 + 创意文创礼品，合称 ToC 主线。

- 用户：C 端家庭用户（家长给孩子、成人自购/送礼）
- 付费结构：硬件买断 + 内容/字体素材包订阅或按次购买
- 客户端：微信小程序优先

架构需为教培 ToB 预留归属树扩展空间，不阻塞但不优先。

明确不做的方向：

- 不做私有化白标授权
- 不做对外开放 API 的云控平台
- 不在主协议里为上两项预留字段

### 1.2 商业假设反推出的架构约束

以下约束在后续章节中具化，在此先固定：

- 根身份：以微信 `unionid / openid` 为账号主键
- 业务数据单一落点：账号、设备归属、任务、内容授权只落在 `manager-api` 的持久化存储
- 设备通道无状态：`xiaozhi-server` 只承载设备级会话与实时流，不持久化业务数据
- 设备可转赠：设备绑定关系必须可迁移
- 内容资源域独立：字体、字帖、画作素材是独立资源域，与设备协议解耦
- 离线可降级:`home / move_abs / 已缓存字帖` 在断网时仍可执行，付费内容允许降级
- 多端一致性：一台设备可被多个客户端同时订阅，写操作按"主控优先"仲裁

### 1.3 不纳入 v2 的商业化议题

不由架构文档承担、延后决定的部分：

- 具体定价
- 内容包形态（按字帖/按字体/按套餐）
- 是否做独立 App
- 渠道与履约

## 2. 角色与职责表

v1 §1 的三行表扩展为五行：

| 角色 | 载体 | 核心职责 | 明确不负责 |
|---|---|---|---|
| `Client` | 微信小程序 `manager-mobile`，后续可扩 Web/App | 身份与订阅中继：登录、设备绑定、选设备、订阅状态、提交高层意图；不承载任务执行数据通路 | 不直接与设备通道对话；不持有任务编排规则；不懂设备协议细节；不作为任务数据必经节点 |
| `BusinessServer` | `manager-api`（Java/Spring） | 账号与鉴权；设备归属与激活码；任务持久化与生命周期；内容资源与授权；客户端订阅转发 | 不承担设备实时通道；不直接编码设备协议；不做实时音视频转发 |
| `DeviceServer` | `xiaozhi-server`（Python） | 设备会话与实时流；语音 ASR/LLM/TTS；与 `U8` 的双向长连接；把设备事件投递给 `BusinessServer` | 不持久化业务数据；不维护账号/归属；不直接对客户端鉴权 |
| `U8 / AI_MCU` | 设备侧 ESP32-S3 | 采集语音/图像/按键；与 `DeviceServer` 保持长连接；把任务转成 `U1` 私有协议；回传状态 | 不做运动闭环；不做业务编排；不暴露 Grbl 方言给上层 |
| `U1 / MOTOR_MCU` | 设备侧 ESP32-S3 | 运动执行、归零、限位、舵机、激光、传感器；设备状态机；安全裁决 | 不理解自然语言；不管理云端会话；不做跨任务编排 |

总原则：

- `Client` 负责 "用户想做什么"
- `BusinessServer` 负责 "这件事能不能做、属于谁、怎么记账"
- `DeviceServer` 负责 "现在怎么和这台设备说话"
- `U8` 负责 "怎么翻译给 U1"
- `U1` 负责 "怎么安全执行"

## 2bis. 数据流向与小程序的定位

小程序定位是**身份/订阅中继**，不是数据中继。为了避免歧义，本节穷举四种典型场景，说明小程序在每条数据路径上处于什么位置。

### 场景 1：用户登录与设备绑定

```
小程序 ──────► BusinessServer
  登录码              发 JWT；写入 binding(account, device)
```

- 小程序在路径上
- 是身份中继的核心场景

### 场景 2：小程序下发"写字"任务

```
小程序 ──► BusinessServer ──► DeviceServer ──► 小智(U8) ──► U1
  write_text       投影成              motion_task       PATH_*
  (text, font)     segments            （设备 cap）       （私有协议）
```

- 小程序在路径起点
- 但"任务生命周期"主体在 BusinessServer 持久化层，不在小程序
- 小程序关掉，任务继续执行

### 场景 3：语音触发"画一只猫"

```
小智(U8) ──► DeviceServer ──► BusinessServer ──► DeviceServer ──► 小智(U8) ──► U1
 语音流         ASR+LLM            生成图形/路径          motion_task         PATH_*
             得出意图            规范化为 segments       派发给设备
```

- **小程序完全不在路径上**
- 这是"小智自己可以画画"的能力：语音入口 → 云端生成图形 → 设备执行
- 小程序只作为旁观者通过订阅看到任务在跑

### 场景 4：小程序查看设备/作业状态

```
U1 ──► 小智(U8) ──► DeviceServer ──► BusinessServer ──► 小程序
 status    motion_event     推送事件         WSS 广播       渲染 UI
```

- 事件流**真正的源头在 U1**，一路上推
- 小程序在路径末端
- 这是"订阅中继"的典型场景：小程序订阅关系登记在 BusinessServer，但数据不经过小程序再回来，是 BusinessServer 直接推给小程序

### 总结

- 小程序**一定在**：身份建立、订阅登记、用户手动下发任务
- 小程序**不在**：语音触发任务、设备执行过程、状态事件回流（除了最末端展示）
- 小程序**永远不**：直连小智、直连设备、持久化任务数据、承载语音流

### 一条关键边界

小程序关掉，以下事情**不受影响**：

- 正在执行的任务继续执行
- 小智继续上报状态到 BusinessServer
- BusinessServer 继续落库
- 语音触发的任务继续可工作

小程序关掉，以下事情**受影响**：

- 实时状态在那一端看不到了（重新打开、`since_seq` 补齐）
- 用户无法从小程序手动下发新任务（但可以语音）

这条边界保证小程序是"可选增强层"，而不是"必选链路节点"。去掉小程序，设备+语音+BusinessServer 仍然是完整系统；加上小程序，用户获得可视化与手动控制。



### 3.1 Capability 原则（继承 v1 并扩展）

- `BusinessServer` 面向两类 capability 编程：
  - 对 `Client` 暴露 `客户端 capability`
  - 对 `DeviceServer` 下发 `设备 capability`
- `DeviceServer` 只面向 `设备 capability`，不认识业务语义
- `U8` 是 `DeviceServer` 与 `U1` 之间的协议适配层
- `U1` 是执行层与安全裁决层

### 3.2 分层（v1 §2.2 扩展）

按四层抽象组织，禁止跨层直连：

- `L4` 会话层：登录、绑定、订阅、鉴权
- `L3a` 客户端语义层：用户意图，例如"写一首诗"、"画这个图案"
- `L3b` 设备语义层：`write_text / draw_generated / draw_asset` 等面向机器的高层任务
- `L2` 设备无关能力层：`home / move_abs / run_path / get_status`（v1 §4 沿用）
- `L1` 设备执行层：`HOME / MOVE / PATH_*` 私有串口命令（v1 §13 沿用）

禁止项（在 v1 基础上补充）：

- `Client` 不直连 `DeviceServer`
- `Client` 不直接生成 `segments`
- `DeviceServer` 不直接操作业务持久化
- `BusinessServer` 不生成 `G-code`
- `U1` 不接收 `L3` 语义

### 3.3 数据所有权分层

- 账号数据：`BusinessServer` 专属
- 设备归属数据：`BusinessServer` 专属
- 任务元数据（谁发起、何时、结果）：`BusinessServer` 专属
- 任务实时状态（running/progress/position）：`DeviceServer` 生成，不落库，仅转发
- 设备端瞬时状态（Grbl 内部）：`U1` 专属，不外泄
- 内容资源（字体/字帖/素材）：`BusinessServer` 专属
- 语音/视觉媒体流：`DeviceServer` 处理，不落业务库

## 4. 系统拓扑

```
+--------+   WSS/HTTPS    +----------------+   内部RPC    +---------------+   WSS(音频+信令)   +-----+   UART @{json}\n   +-----+
| Client |  <----------> | BusinessServer |  <--------> |  DeviceServer  |  <------------->  | U8  |  <------------->   | U1  |
| 小程序 |                |  manager-api   |             | xiaozhi-server |                    |     |                    |     |
+--------+                +----------------+             +----------------+                    +-----+                    +-----+
     Edge-A                       Edge-B                      Edge-C                             Edge-D
```

- `Edge-A`：`Client ↔ BusinessServer`
- `Edge-B`：`BusinessServer ↔ DeviceServer`
- `Edge-C`：`DeviceServer ↔ U8`
- `Edge-D`：`U8 ↔ U1`（即 v1 §12~§18 定义的私有协议，v2 沿用）

## 5. 四条通信边界

### 5.1 Edge-A：Client ↔ BusinessServer

- 传输：HTTPS + WebSocket over TLS（小程序原生支持）
- 通道分工：
  - 短连接 HTTPS：登录、绑定、列表查询、下发任务（幂等提交）
  - 长连接 WSS：订阅设备状态、作业进度、错误事件
- 鉴权：登录后下发 JWT，WSS 连接首帧携带
- 消息风格：`面向客户端 capability`
- 禁止项：不在此边界暴露 `Grbl` / `U1 私有协议` / `DeviceServer 内部会话`

### 5.2 Edge-B：BusinessServer ↔ DeviceServer

- 传输：内部 HTTP + 内部事件总线（具体技术延后定，但边界形式先固定）
- 方向 1：`BusinessServer → DeviceServer` 下发 `设备 capability`
- 方向 2：`DeviceServer → BusinessServer` 上报 `设备事件`（含设备在线、任务状态、错误）
- 鉴权：内部服务互信（mTLS 或共享密钥），不共用 Client 的 JWT
- 禁止项：`DeviceServer` 不直接访问 `BusinessServer` 的业务持久化层，只能通过 HTTP 接口调用

### 5.3 Edge-C：DeviceServer ↔ U8

- 传输：WebSocket over TLS（沿用 xiaozhi-server 现状）
- 通道内容：
  - 语音流（上行音频、下行 TTS）
  - 会话信令（激活、心跳、版本）
  - 任务帧（`设备 capability`）
  - 事件帧（status / result / error / progress）
- 鉴权：设备激活码 + 一次性会话 token
- 禁止项：此边界不直接承载 `Client` 订阅流；小程序要看状态必须走 Edge-A

### 5.4 Edge-D：U8 ↔ U1

沿用 v1 §12~§18 的私有串口协议：

- 帧形式：`@{json}\n`
- 命令集：`HOME / MOVE / PATH_BEGIN / PATH_SEG / PATH_END / GET_STATUS / PAUSE / RESUME / STOP / ESTOP`
- 响应类型：`ack / status / result / error`
- 字段：`msg_id / task_id / state / error_code / alarm_code / progress / position`

v2 对 Edge-D 不改协议，仅对实现提要求（见 §15 实施清单）。

## 6. 标准任务模型（双向）

v1 §3 的任务模型只覆盖下行。v2 明确双向模型。

### 6.1 下行：任务下发

沿用 v1 §3 外层字段，新增 `owner` 与 `client_origin`：

```json
{
  "task_id": "uuid",
  "task_type": "motion",
  "source": "voice|button|client",
  "target": "u1",
  "owner": {
    "unionid": "oXxx",
    "device_id": "dev_xxx"
  },
  "client_origin": {
    "client_id": "wx_mp",
    "request_id": "uuid"
  },
  "capability": "home|move_abs|run_path|get_status|write_text|draw",
  "params": {},
  "constraints": {
    "requires_homing": true,
    "timeout_ms": 30000,
    "safety_level": "strict"
  }
}
```

`target` 取值扩展：

- `u1`：最终下到设备执行
- `device`：由 `DeviceServer` 处理（例如纯会话控制）
- `business`：由 `BusinessServer` 处理（例如任务持久化查询）

### 6.1bis 幂等

网络抖动或客户端重试可能让同一意图被重复提交。幂等规则：

- `client_origin.request_id` 是幂等键，由客户端生成
- `BusinessServer` 对 `(account_id, device_id, request_id)` 三元组去重
- 去重窗口 24 小时，窗口内重复请求返回首次 `task_id`，状态字段返回当前实际状态
- 语音意图的 `request_id`（§6bis.3）同样规则
- `client.cancel_task / pause_task / resume_task` 本身幂等，不依赖 `request_id`

### 6.2 上行：事件订阅

上行事件统一结构：

```json
{
  "event_id": "uuid",
  "event_type": "device_status|job_status|job_progress|job_result|job_error|device_online",
  "device_id": "dev_xxx",
  "task_id": "uuid|null",
  "ts": 1730000000000,
  "seq": 123,
  "payload": {}
}
```

`seq` 在同一 `device_id` 内单调递增，用于客户端断线重连后补齐。

### 6.3 任务生命周期五态

`BusinessServer` 统一对外呈现五态，与 v1 §15 的设备结果映射：

| 业务态 | 触发来源 | 对应 U1 状态/结果 |
|---|---|---|
| `accepted` | `U1` 返回 `ack.accepted=true` | 进入 `HOMING/RUNNING` 前的瞬态 |
| `running` | `U1` `status: RUNNING/HOMING/PAUSED` | 执行中 |
| `done` | `U1` `result: DONE` | 成功完成 |
| `failed` | `U1` `error` 或超时 | 失败，含错误码 |
| `cancelled` | 客户端或系统发起 `STOP` 且成功 | 受控中止，对应 `E007` |

硬约束：

- `STOP` 与 `ESTOP` 语义严格分离，禁止混用
- `STOP` 是受控停止，目标是让设备在仍可控的前提下安全停到 `IDLE`，其业务结果只能投影为 `cancelled`
- `ESTOP` 是最高优先级安全中断，目标是立即打断执行并进入设备态 `ESTOP`，其业务结果只能投影为 `failed + E_ESTOP`
- `PAUSE/RESUME/STOP/ESTOP` 在 U1 内部可以映射为不同 Grbl/底层控制手段，但这些手段只是实现细节，不得反向改变上层状态语义

## 6bis. 语音意图如何流向任务

§2bis 场景 3 画了语音触发任务的整链，本节补足"语义在哪一层落地"。

### 6bis.1 意图分流原则

`DeviceServer` 持有 ASR 与 LLM，输出意图。意图分两类：

- `chat_intent`：闲聊、问答、查天气等，闭环在 `DeviceServer` 内部，不经过 `BusinessServer`
- `device_intent`：与设备能力相关的意图（写字、画画、归零、暂停），必须经过 `BusinessServer` 投影并鉴权

判定规则放在 `DeviceServer` 的意图理解阶段，按白名单触发：

- 命中白名单 capability 关键词 → `device_intent`，调用 §6bis.3 接口
- 未命中 → `chat_intent`，沿用既有小智链路

### 6bis.2 为什么必须绕回 BusinessServer

- 鉴权：用户是否有权用这只字体/这套素材，账号体系在 `BusinessServer`
- 投影：`write_text → segments` 的字体资源在 `BusinessServer`
- 持久化：每次任务必须落库以便客户端订阅、查询、计费
- 安全：设备执行前的工作区裁剪、参数校验集中在 `BusinessServer`

直接让 `DeviceServer` 自己投影会复制账号/资源/任务表三套数据，违反 §3.3 数据所有权分层。

### 6bis.3 DeviceServer → BusinessServer 的意图回调

新增一条 Edge-B 反向接口：

```json
POST /internal/v1/intent_submit
{
  "device_id": "dev_xxx",
  "source": "voice",
  "intent": "write_text|draw|home|move_abs|pause|resume|stop|estop",
  "params": { "text": "你好", "font_hint": "kai" },
  "session_id": "voice_session_xxx",
  "request_id": "uuid"
}
```

`BusinessServer` 的处理顺序：

1. 用 `device_id` 反查绑定账号，得到 `account_id`
2. 用 `account_id` 校验 `entitlements`
3. 通过则等同于客户端 `submit_*`，投影成 `segments` 后下发
4. 不通过则 `BusinessServer` 通过 Edge-B 回 `DeviceServer` 一条 `tts_hint`，由 `DeviceServer` 合成语音回复"这个字体还没解锁"

### 6bis.4 语音任务的可见性

语音触发的任务和小程序触发的任务在 `BusinessServer` 完全等价：

- 同样落 `tasks` 表
- 同样产生 §6.2 的事件流
- 已订阅该设备的客户端会看到 `source=voice` 的新任务出现

这保证了"用户语音让小智画画时，家长的小程序能看到孩子在画什么"的体验闭环。

### 6bis.5 边界纪律

- `DeviceServer` 不调用 `BusinessServer` 的业务持久化层，只调用 §6bis.3 这条接口
- `DeviceServer` 不直接给 `U8` 下发 `motion_task`，只通过 `BusinessServer` 派回的标准任务通道
- `BusinessServer` 不直接和设备说话，永远经过 `DeviceServer`

## 6ter. 语音鉴权与杂音过滤

§6bis 解决了"语音意图怎么变成任务"，但没解决"谁说话才算数"。本节定义语音入口的鉴权与抗噪机制。

### 6ter.1 风险面

- **陌生人控制**：陌生人对设备说话即可发起任务
- **被动触发**：电视、广告、家人闲聊里的命令式语句被 ASR 误识为意图
- **环境本底**：空调、风扇、远处人声被 ASR 拼出虚假命令
- **重放攻击**：录音回放冒充已注册声纹（v2 不要求强防御，但要登记风险）

### 6ter.2 分层裁决：四道关

```
麦克风 ──► [1.VAD]──► [2.唤醒词]──► [3.声纹]──► [4.意图分流]──► BusinessServer 安全裁决(§10bis)
            过杂音      过随机话语    过非家庭成员   过非命令意图
```

每一道关对应一种过滤目的，**串联而非冗余**：

| 关 | 位置 | 作用 | 失败处理 |
|---|---|---|---|
| 1. VAD | U8 设备端 | 过滤无人声段、低信噪比段 | 静默丢弃 |
| 2. 唤醒词 | U8 设备端 | 过滤未带"小智"前缀的对话 | 静默丢弃 |
| 3. 声纹 | DeviceServer | 过滤未注册成员的语音 | 进入"陌生人模式"，见 §6ter.6 |
| 4. 意图分流 | DeviceServer | 过滤闲聊；命令转 BusinessServer | 闲聊回复不下任务 |

### 6ter.3 VAD 与抗噪

- 设备端使用 ESP-SR 或等效轻量 VAD，门限设为"正常对话距离 50 cm 内可触发，1 m 外日常背景音不触发"
- 唤醒词使用上游 xiaozhi-esp32 既有方案，**唤醒词错误率从严**：建议 1 次/24h 以内
- VAD 通过后再上传音频，从源头抑制无意义流量
- 设备本地保留环境噪声基线，自动调阈值；阈值跨家庭差异大，不在云端统一

### 6ter.4 声纹模型分级

| 档位 | 行为 | 适用场景 |
|---|---|---|
| `voiceprint_off` | 不做声纹比对，任何人都能控制 | 商场演示、公共展示 |
| `voiceprint_loose` | 仅对**写类任务**做声纹校验，闲聊放行 | 默认家庭模式 |
| `voiceprint_strict` | 闲聊与写类都需要声纹通过 | 家长担心电视触发的家庭 |

档位是设备级设置，存于 `device_caps.voiceprint_mode`，由小程序设。默认 `voiceprint_loose`：陌生人能聊，不能让机器写字画画。

### 6ter.5 声纹注册流程

```
小程序                   BusinessServer            DeviceServer            设备
  │                            │                        │                    │
  │  "为孩子注册声音"         │                        │                    │
  │  采集 5~8 秒朗读片段      │                        │                    │
  │ (在小程序本地完成)         │                        │                    │
  │                            │                        │                    │
  │  上传音频片段             │                        │                    │
  │ + member_role(child/parent)│                        │                    │
  │──────────────────────────►│                        │                    │
  │                            │ 调用 ASR/声纹服务      │                    │
  │                            │ 计算嵌入向量          │                    │
  │                            │ 写入 voiceprints       │                    │
  │◄──── ok, member_id ────────│                        │                    │
  │                            │                        │                    │
  │                            │ 把活跃声纹下发给 DeviceServer 缓存          │
  │                            │───────────────────────►│                    │
```

注册片段在 BusinessServer 完成嵌入计算后**立即销毁原始音频**，只保留嵌入向量。不在设备上做声纹注册，避免低算力误判和被攻击。

### 6ter.6 实时比对

- 每次唤醒后的语音片段在 DeviceServer 计算嵌入向量
- 与该 `device_id` 绑定的全部成员声纹比对
- 取最高相似度，超过阈值视为命中；未命中视为"陌生人"
- 阈值默认 0.78（具体数值随声纹模型微调），`voiceprint_strict` 档位提到 0.85

陌生人模式：

- `voiceprint_loose`：陌生人可闲聊（DeviceServer 内闭环），不可发起 §6bis.3 的 `intent_submit`
- `voiceprint_strict`：陌生人语音直接静默丢弃，TTS 也不回复
- 任何陌生人触发尝试落到 `safety_audit`（§10bis.11），便于家长复盘"谁试图让机器写什么"

### 6ter.7 家庭成员管理

`members(member_id, account_id, device_id, role, voiceprint_vec, registered_at)`

- 一台设备可有多个成员（爸爸/妈妈/孩子）
- 角色 `parent | child | other`
- 角色不影响命令权限（任何已注册成员都能下命令），但用于审计与未来分级（例如 `child` 不能解锁激光）
- 小程序内可"重录声纹""删除成员"
- 转赠（§11.5）时全部声纹一并清空

### 6ter.8 儿童使用兜底

声纹模型对儿童识别误差较大（声音变化快）：

- 儿童成员每 6 个月自动提示重录
- 失败超过 3 次后自动降级当前会话为 `voiceprint_off`，并通过小程序事件通知家长
- 家长可在小程序"以家长身份代孩子下任务"，绕过声纹（属于显式越权，落审计）

### 6ter.9 隐私

- 原始音频片段：注册阶段计算完嵌入立即销毁；使用阶段不落库
- 嵌入向量：仅存于 BusinessServer，不下发到设备 NVS
- DeviceServer 缓存的活跃声纹在设备解绑或转赠时立即清除
- 声纹数据视为生物识别信息，遵循中国个人信息保护法 PIPL 中"敏感个人信息"处理要求；小程序在注册前必须有显著告知与单独同意

### 6ter.10 不依赖声纹的兜底

声纹有可能失败、误判、暂不可用。以下兜底**永不依赖声纹**：

- §10bis 设备侧安全裁决永远独立生效
- 物理急停按钮永远独立生效
- 小程序的"主控会话"始终高于语音通道（§12.2），主控明确反对的任务，无论声纹通过与否都不执行

### 6ter.11 不做的事

- 不做声纹解锁付费内容（避免误判带来的客诉）
- 不做声纹反向用作账号登录凭据（生物识别用作登录的合规复杂度过高）
- 不做"识别情绪"等衍生分析
- 不做实时音频本地落盘（家庭隐私场景敏感度高）

## 8. 客户端 Capability

按操作目的分四组。所有命名以 `client.` 前缀，便于与设备 capability 区分。

### 8.1 身份与设备

- `client.login`：微信登录，换取 JWT
- `client.list_devices`：列出当前账号已绑定的设备
- `client.bind_device`：用激活码 + 设备序列号绑定一台设备
- `client.unbind_device`：解绑（用于转赠场景）
- `client.transfer_device`：把设备从当前账号转移到另一账号（双方确认流）

### 8.2 任务下发

- `client.submit_write_text`：提交写字任务
- `client.submit_draw`：提交生成式画图任务
- `client.submit_motion`：低层运动（home/move_abs），仅高级模式可见
- `client.cancel_task`：取消任务
- `client.pause_task` / `client.resume_task`：暂停/恢复

### 8.3 任务与历史

- `client.list_tasks`：分页列出当前账号在指定设备上的任务
- `client.get_task`：拉取某任务的全量元数据
- `client.list_recent_events`：拉取最近事件，用于断线后的补齐

### 8.4 实时订阅

- `client.subscribe_device`：订阅指定设备的状态流
- `client.subscribe_task`：订阅某个任务的进度流
- `client.unsubscribe_*`：解除订阅

订阅一律走 Edge-A 的 WSS 通道。

## 8bis. 小程序 UI 模板约束

小程序是产品脸面，不是后台。UI 由模板而非自己拼装承担基础观感。

### 8bis.1 强制原则

- 禁手搓页面骨架、组件库、主题、图标体系
- 必须基于 GitHub 上具有视觉完成度的现成模板二次开发
- 选定模板前不得提交"自定义页面"，不得为图省事先用占位 UI 上线再"以后再美化"
- 任何"以后再换模板"的承诺等同于不会发生

理由：

- 美工资源有限，自研观感稳定低于精选模板
- 模板提供的不仅是页面，还有交互细节、动效、空状态、错误态，这些自研最容易省略
- 一次性选好模板，避免后期推倒重来

### 8bis.2 选型标准

候选模板必须同时满足：

- 平台兼容：支持 `uni-app + mp-weixin`，或可低成本迁移
- 视觉成熟度：作者交付了亮色/暗色、空状态、加载、错误态完整套件
- 维护活跃：近一年内有提交
- 协议合规：MIT / Apache-2.0 / BSD 等可商用协议；GPL 类禁用
- 组件齐备：列表、表单、详情、权限引导、订阅状态、进度条样式至少一份成例
- 体积可控：模板包对小程序 2 MB 上限友好，必要时按页拆包

候选优先级（从高到低，仅作起点，最终由实施时回查 GitHub 当年活跃度决定）：

- `uni-app` 体系内的多端 UI 模板，如管理后台/商城/智能硬件类的成熟仓库
- 微信小程序原生模板，但需评估迁移到 `uni-app` 的成本
- 跨平台 UI 库 + 模板组合（例如 uView Plus、wot-ui 等），适合在它们之上做主题改色

### 8bis.3 改造边界

允许：

- 替换品牌色与字体
- 替换图标包
- 删除模板中本项目用不到的页面
- 在模板组件之上做业务封装（例如把模板的"订单详情页"改为"任务详情页"）

禁止：

- 大幅重写模板的导航结构，导致丢失模板的视觉一致性
- 把多个模板的页面拼凑混用
- 私自引入第三方组件库覆盖模板原生组件，导致主题断裂

如果改造范围超过 30% 页面，应视为"模板不合适"重新选型，而不是继续魔改。

### 8bis.4 与架构的关系

模板是 `Client` 角色的实现细节，但其选择影响：

- §8 客户端 Capability 的页面承载形态
- §9 订阅事件的 UI 呈现（要求模板有"实时刷新型列表/卡片"成例）
- §14bis 故障恢复的引导界面（要求模板有清晰的"操作引导/向导"成例）
- §10bis.7 / §10bis.8 安全提示与确认对话（要求模板有完整的二次确认 UI）

选型时必须把上述四类场景在模板里走一遍 demo，确认各场景都有对应的现成组件，再敲定。

### 8bis.5 选型记录

模板选定后，在 `firmware/.../docs` 或 `server/.../docs` 下登记一份 `ui-template.md`：

- 模板仓库地址
- 选用版本号或 commit
- 协议
- 替换记录（颜色、字体、图标）
- 砍掉的页面清单
- 不与模板兼容的业务点

这条记录在每次升级模板时同步更新。没有此记录的 PR 不允许并入主分支。

### 8bis.6 出现在 §19 推送门禁中的对应条款

在 §19.2 推送门禁加一条隐含规则：

- Client 侧 PR 涉及任何页面级 UI 时，必须有 `ui-template.md` 中的来源指向；纯手搓页面骨架的 PR 不并入

## 9. 实时订阅协议

### 9.1 通道

- 协议：WSS over TLS
- 路径建议：`/ws/v1/client`
- 鉴权：连接首帧发送 `auth` 消息携带 JWT
- 心跳：客户端 30 秒心跳，服务端回 pong；2 次未回视为断线

### 9.2 帧格式

客户端 → server：

```json
{ "op": "subscribe_device", "device_id": "dev_xxx", "req_id": "uuid" }
{ "op": "subscribe_task",   "task_id":  "uuid",    "req_id": "uuid" }
{ "op": "unsubscribe",      "topic":    "device:dev_xxx" }
{ "op": "ack",              "ack_seq":   123 }
{ "op": "ping" }
```

server → 客户端：

```json
{ "type": "subscribed", "topic": "device:dev_xxx", "since_seq": 123 }
{ "type": "event",      "event": { ...见 §6.2 } }
{ "type": "pong" }
{ "type": "error",      "code": "E_AUTH|E_NOT_FOUND|E_FORBIDDEN", "message": "..." }
```

### 9.3 断线补齐

- 每个 `device_id` 维持一个 `seq` 单调递增计数
- `BusinessServer` 在内存窗口里缓存最近 N 条事件（建议 200 条）
- 客户端重连 `subscribe_device` 时携带 `since_seq`，server 把窗口内大于 `since_seq` 的事件回灌
- 超出窗口的，让客户端走 `client.list_recent_events` 拉取

### 9.4 订阅鉴权规则

- `subscribe_device`：要求当前账号是该设备的绑定者
- `subscribe_task`：要求当前账号是该任务的发起者或目标设备的绑定者
- 任意越权订阅一律返回 `E_FORBIDDEN` 并断开

### 9.5 多端一致性

- 一个账号可以多端同时订阅同一设备
- 写操作（任务下发、取消、暂停）以"主控会话"优先：
  - 主控会话由 `client.bind_device` 时记录的最近一次 `client_id` 担任
  - 非主控发起写操作时返回 `E_NOT_PRIMARY`，但允许查看
  - 主控可以通过 `client.claim_primary` 转移

## 10. write_text / draw_generated 投影管线

### 10.1 责任归属

字形与生成图形离散化由 `BusinessServer` 完成。理由：

- 字体、字帖是商业内容资产，必须在持有账号体系的一侧
- 字体文件大、版权敏感，不适合放到 `DeviceServer` 或设备
- 修改字体/算法时不需要重新发布设备固件

`DeviceServer` 不识别 `write_text / draw_generated / draw_asset` 的投影细节，只识别 `run_path`。

### 10.2 输入

`client.submit_write_text` 输入：

```json
{
  "device_id": "dev_xxx",
  "text": "你好世界",
  "font_id": "kai_v1",
  "canvas": {
    "origin_mm": [10.0, 10.0],
    "width_mm": 180.0,
    "height_mm": 120.0,
    "line_height_mm": 28.0,
    "font_size_mm": 22.0,
    "align": "left|center"
  },
  "feed_rate": 1200,
  "pen_mode": "pen"
}
```

`client.submit_draw` 输入：

```json
{
  "device_id": "dev_xxx",
  "prompt": "一只简笔小猫",
  "image_ref": null,
  "svg_text": null,
  "canvas": { "origin_mm": [10.0, 10.0], "scale": 1.0 },
  "feed_rate": 1200,
  "pen_mode": "pen"
}
```

### 10.3 投影管线

```
write_text(text, font_id, canvas)
    │
    ▼ BusinessServer.text_layout            排版：确定每个字位置
    │
    ▼ BusinessServer.glyph_resolve          查字形：每字 → 矢量轮廓
    │
    ▼ BusinessServer.discretize             离散化：曲线 → M/L 段
    │
    ▼ BusinessServer.canvas_transform       变换：字体坐标 → canvas_mm
    │
    ▼ BusinessServer.task_persist           任务落库：拿到 task_id
    │
    ▼ Edge-B 下发                          run_path { segments, feed_rate }
    │
    ▼ DeviceServer → U8                    打包为 capability 任务
    │
    ▼ U8 编码                              PATH_BEGIN/PATH_SEG/PATH_END
    │
    ▼ U1                                   执行
```

`draw_generated(prompt|image|svg, canvas)`
    │
    ▼ BusinessServer.graphic_planner       生成或接收图形输入
    │
    ▼ single_line_normalizer               约束为单线、无填充、黑白简笔 SVG
    │
    ▼ BusinessServer.discretize            离散化：曲线 → M/L 段
    │
    ▼ stroke_planner                       节点简化、路径排序、断笔规划
    │
    ▼ safety_validator                     工作区/段数/总长度/预计耗时校验
    │
    ▼ BusinessServer.task_persist          任务落库：拿到 task_id
    │
    ▼ Edge-B 下发                          run_path { segments, feed_rate }
    │
    ▼ DeviceServer → U8 → U1               执行
```

### 10.4 离散化与坐标系约定

- 内部统一使用 `canvas_mm` 坐标系，原点在画布左下角
- `BusinessServer` 输出的 `segments` 与 v1 §4.3 定义的 `M/L` 一致
- 曲线在 `discretize` 阶段近似为折线，弦高公差不大于 0.05 mm
- 单段长度上限 5 mm，避免单段过长导致进度上报粗糙
- `BusinessServer` 必须做工作区裁剪：超出 `device_caps.workspace_mm` 的段直接拒绝任务，返回 `E002 OUT_OF_RANGE`，不做静默裁剪

### 10.5 字体、生成图与 fallback 资源

- 资源域：`BusinessServer` 持有 `fonts / ai_plans / assets` 三类资源表
- 主路线：`draw_generated` 只接受或产出单线、无填充、黑白简笔 SVG，不以预置 `asset` 作为前提
- fallback：`draw_asset` 仅用于 demo、离线兜底、默认免费集合
- 授权：每个账号有 `entitlements`，记录可用字体、fallback 素材与增值生成能力
- 缓存策略：投影结果可缓存 `(font_id, text_hash)`、`(prompt_hash, model_tier, params_hash)` 或 `(asset_id, params_hash)`，命中直接复用 `segments`
- 离线降级：客户端可在小程序本地缓存最近一份 `segments` 结果，断网时由小程序直接调 `client.submit_motion + run_path`（仍走 BusinessServer 鉴权，但不依赖重新生成）

### 10.6 进度语义

- `BusinessServer` 把 `progress ∈ [0,1]` 直接透传给客户端，不做加权
- `progress` 由 `U1` 按 `已完成段数 / total_segments` 产生
- 客户端展示时可叠加 `task_id` 的总段数与已耗时

## 10bis. 安全裁决与前置校验

写字机/雕刻机带有真实力学输出（运动、激光、舵机），任何执行前的校验缺失都会变成机械损伤或人身风险。本节定义"任务真正下到 U1 之前的最后一道关"。

### 10bis.1 双重裁决原则

安全裁决在两层独立做，不允许任意一层兜底：

- `BusinessServer` 侧：基于已知设备能力做**前置裁决**，是大多数错误的拦截层
- `U1` 侧：基于实时硬件状态做**最终裁决**，是不可绕过的兜底层

两层规则**不重叠简化**：上层校验失败立即拒绝；上层放行后 U1 仍可独立拒绝。`U1` 永远是安全决策的最终所有者。

理由：

- 上层缓存的设备能力、坐标、状态都可能过期或被绕过
- 限位、急停、机械过载只有 U1 能实时观测
- 双层独立避免"上层放行=安全"的错误信任传递

### 10bis.2 设备能力数据 device_caps

`BusinessServer` 持有每台设备的能力快照表 `device_caps(device_id, ...)`：

- `workspace_mm`：`{x_min, x_max, y_min, y_max, z_min, z_max}`，物理工作区
- `safe_margin_mm`：工作区内边距，所有任务的可写区收缩这个值
- `home_position_mm`：归零完成后的标称坐标
- `home_tolerance_mm`：判定"在原点"的容差球半径
- `max_feed_rate`：进给速率上限
- `pen_clearance_mm`：抬笔安全高度
- `requires_homing_for`：必须先归零的 capability 集合，默认 `{move_abs, run_path, write_text, draw}`

数据来源：

- 工厂默认值：出厂阶段固化（`firmware/u1-grbl/.../dlc_motor_control_p1.h`）
- 校准更新：`U1` 主动上报或客户端校准流程触发更新

`device_caps` 是 BusinessServer 自己的数据，不是 U1 的实时状态镜像。

### 10bis.3 设备实时状态 device_runtime_state

由 U1 上报、DeviceServer 转发、BusinessServer 缓存的实时切片：

- `state`：`IDLE / RUNNING / HOMING / PAUSED / ALARM / ERROR / ESTOP`
- `homed`：真实归零状态（每个轴独立位）
- `position_mm`：当前坐标 `{x, y, z}`
- `at_home`：派生字段，`homed && |position - home_position| < home_tolerance`
- `last_status_ts`：最近一次更新时间戳

时效约束：

- `last_status_ts` 早于 5 秒视为"过期"
- 过期时 BusinessServer 必须先发 `get_status` 刷新，再做安全决策
- 过期超过 30 秒视为设备掉线（§12bis）

### 10bis.4 capability 前置校验清单

每个写类 capability 在 BusinessServer 必须经过以下校验，不通过直接返回业务错误码，**不下发**到 DeviceServer：

| capability | 必校项 | 失败错误码 |
|---|---|---|
| `home` | 设备非 `RUNNING/HOMING`；非 `ESTOP` | `E_DEVICE_BUSY / E_ESTOP` |
| `move_abs` | `homed=true` 全轴；目标 ∈ 工作区扣减 `safe_margin`；feed_rate ≤ `max_feed_rate`；设备 `IDLE` | `E_NOT_HOMED / E_OUT_OF_RANGE / E_INVALID_PARAM / E_DEVICE_BUSY` |
| `run_path` | 同 `move_abs` 全部条件；`segments` 全部坐标点 ∈ 可写区；段间最大跳跃合法；总段数上限 | 同上 + `E_INVALID_PARAM` |
| `write_text` | 投影后等同 `run_path` 的全部检查；字体已授权 | 同上 + `E_NOT_ENTITLED` |
| `draw_generated` | 先满足单线、无填充、节点数/总长度/预计耗时上限；投影后等同 `run_path` | 同上 + `E_INVALID_DRAWING` |
| `draw_asset` | 同 `write_text` | 同上 |
| `pause / resume / stop` | 设备处于可暂停态 | `E_INVALID_STATE` |
| `estop` | 总是允许下发 | 无 |

补充约束：

- "在原点"是一个**强于** `homed` 的状态。`homed=true` 仅表示坐标系已建立，不代表当前位置就在原点。`at_home` 派生字段才反映真实在原点
- 默认所有写类 capability 不要求 `at_home=true`，仅要求 `homed=true`。但客户端可以传 `constraints.require_at_home=true` 强制要求"必须在原点才能开始"
- `write_text / draw` 投影成 `run_path` 后，校验**也对投影结果生效**，不只对原始参数生效。这避免"参数合法、投影出来超界"的漏洞

### 10bis.5 工作区与可写区

定义：

- 工作区 `workspace`：物理可达坐标范围
- 可写区 `writable` = 工作区 ∪ 各轴向内收缩 `safe_margin_mm`

所有用户路径必须落在可写区。`safe_margin` 默认值（建议）：

- `x / y`：5 mm
- `z`：1 mm（避免过度下压）

收缩理由：

- 工作区边界是机械极限，紧贴边界容易触发限位 `E005`
- 留出余量让小误差不致中断任务

### 10bis.6 运行期监控

任务执行期间，U1 持续监控以下条件，违反即进入 ALARM 或 ESTOP：

- 限位开关触发 → `E005 LIMIT_TRIGGERED`
- 实时坐标超出 `workspace`（软件限位） → `E002 OUT_OF_RANGE`，软件限位由 v1 §22 `DEFAULT_SOFT_LIMIT_ENABLE = 1` 已开启
- 物理急停按钮触发 → `E008 ESTOP_TRIGGERED`
- 步进丢步检测（如未来引入闭环）→ `E_INTERNAL`

BusinessServer 的前置校验**不替代**运行期监控。只要 U1 实时观测到任何越界，立即中止当前任务并上报，无论 BusinessServer 有没有放行。

### 10bis.7 笔/激光保护

写字机/激光会留下不可逆痕迹，需要额外保护：

- 任何任务开始前，U1 必须确认 Z 轴在 `pen_clearance_mm` 抬笔位
- 任务被 `pause` 时，U1 必须自动抬笔，避免静止时墨水/激光持续作用
- `cancel/abort/estop` 后 U1 必须自动抬笔
- `resume` 时先抬笔移到下一段起点，再下笔，不允许"原地落笔继续"

这部分在 v2 是 U1 实现要求，BusinessServer 不参与。BusinessServer 只通过 `pen_mode` 字段告诉 U1 当前是 `pen / laser / lift_only`，由 U1 决定具体动作。

### 10bis.8 激光使用的额外门禁

激光会造成视觉损伤与火灾，比写字更敏感：

- 默认配置 `pen_mode != laser`，激光不参与日常写字流程
- 客户端要打开激光必须显式 `pen_mode=laser` + 二次确认
- 当 `pen_mode=laser` 时，BusinessServer 强制要求 `at_home=true` 才允许开始
- v2 不开放激光功率超过出厂上限的能力，`max_laser_power` 由 `device_caps` 限定

### 10bis.9 龙门矫正暂缓

当前硬件结构经机械设计已固定 X/双 Y/Z 几何，v2 阶段**关闭双 Y 差分方正矫正**，但**保留双 Y 同步归零**：

- 关闭：`DEFAULT_HOMING_SQUARED_AXES`（Grbl 的软件方正校正，利用两个限位差分调整）
- 保留：`LIMITS_TWO_SWITCHES_ON_AXES` 中与 Y/Y2 相关的同步归零语义——`Y_LIMIT` 与 `Y2_LIMIT` 都继续参与归零，双 Y 两侧同时触到各自限位后才认为 Y 轴归零完成
- 保留：`Y` 和 `Y2` 各自独立的 STEP/DIR 输出，电气上仍是双电机同步驱动

**禁止两种极端**：

- 不要关闭到"只看 Y 限位、忽略 Y2"——否则龙门两侧不同步，机械结构会别住
- 不要开启 squaring 差分——结构已固定，差分修正反而会引入持续偏斜

此决策与 v1 §22 / `dlc_motor_control_p1.h` 的现状不同，v2 §16.5 将此作为独立实施项。未来若机械松紧出现明显偏差或更换刚度更高的结构再重启差分方正。重启时需同步更新 §10bis.4 的限位规则。

### 10bis.10 安全决策不可被覆盖

任何来源都**不允许**绕过 §10bis 校验：

- 客户端 capability 不开放 `force / unsafe / skip_check` 类参数
- 语音意图（§6bis）走相同的安全裁决，不开"语音特权"
- 内部调试需要时通过独立 `dev_motion` capability，仅 dev 构建可用，并在 `device_caps.dev_mode=true` 时才解锁

这条规则的存在意义：避免商业化迭代中产生"为了用户体验流畅而开后门"的诱惑。

### 10bis.11 安全事件审计

所有安全裁决拒绝事件必须落库，便于复盘：

- BusinessServer 拒绝：写入 `safety_audit(account_id, device_id, capability, reason, ts)`
- U1 拒绝：通过 `error` 事件上报，BusinessServer 镜像写入审计表
- 审计数据保留 90 天

这条不在第一个里程碑必做，但表结构在 §16.2 数据库初始化时就要预留位。

### 10bis.12 急停硬件归属

§10bis.6 / §14bis.2 多次提及"物理急停按钮"。当前硬件方案（`docs/硬件连接与GPIO分配说明.md`）板上**没有独立急停接口**。v2 阶段采用三层兜底，**不设独立急停按钮但保证仍有物理可触达手段**：

| 层 | 触发方式 | 响应路径 | 响应延迟 |
|---|---|---|---|
| 软急停 - 客户端 | 小程序"急停"按钮 | Edge-A → BusinessServer → DeviceServer → U8 → U1 `ESTOP` 命令 | 网络往返，约 200~800 ms |
| 软急停 - 设备本体 | U1 BOOT 按键短按（注：U8 的 BOOT 已用于配网长按，此处指 U1 板上的 BOOT 按键） | U1 直接进入 `ESTOP` 状态 | < 10 ms |
| 硬切断 - 用户 | 拔电 / 关电源开关 | 物理掉电 | 即时 |

约束：

- 软急停 - 客户端：受网络约束，**不可作为唯一手段**，仅适合"看到屏幕里跑偏，赶紧停"
- 软急停 - 设备本体：U1 固件主循环需要在 ms 级响应 BOOT 按键中断，立即停步进 + 抬笔 + 进入 `ESTOP` 状态
- 硬切断：写字机/激光机的最后保险；商品包装内说明书必须显著标注"紧急情况请直接关电源"

明确未做：

- 不设独立急停红色按钮（硬件成本与 PCB 改版代价高于 v2 阶段收益）
- 不假设有独立急停红色按钮（避免文档与硬件不一致）

### 10bis.13 急停升级路径

未来硬件 P2/P3 引入独立急停按钮时：

- 占用 U1 一个未用 GPIO（参考 `硬件连接与GPIO分配说明.md` §4.2 未占引脚清单）
- 走硬件中断而非轮询
- 优先级覆盖一切软件层
- §10bis.10 / §10bis.12 / §14bis.2 同步更新

v2 期间任何文档若提及"独立急停按钮"，应理解为"软急停 + 拔电"组合，不应理解为存在独立硬件按钮。

## 10ter. 内容安全裁决

设备会说话（TTS 输出）、会写字（G-code 输出）、会画画（生成图形 / SVG / 图片输入投影输出）。**任何一路内容**都可能让家长看到后崩溃，也直接决定小程序能否上架。

### 10ter.1 四条内容路径

| 路径 | 来源 | 去向 | 风险 |
|---|---|---|---|
| P1 LLM 输出 | LLM provider | TTS → 扬声器 | 闲聊里夹带不宜内容 |
| P2 ASR 转写 | 用户语音 → ASR | 意图或直接 `write_text` | 用户说什么设备就写什么 |
| P3 `write_text` 客户端提交 | 小程序文本输入 | 投影 → 设备 | 用户手动输入不宜内容 |
| P4 `draw_generated` 输入 | prompt / image / svg / transcript | 投影 → 设备 | 生成内容不合规、非单线 SVG、图形不可执行 |

### 10ter.2 审核分层

- P1：LLM 输出先过"出站审核"，再进 TTS
- P2、P3：文本在 BusinessServer 投影**之前**过"入站审核"，拒绝即任务不落库
- P4：生成输入在进入图形规范化前过"入站审核"；生成结果在落轨迹前过"出图审核"

入站/出站审核使用同一个 `content_moderation` 服务，不重复实现。

### 10ter.3 审核规则来源

- 国家有关互联网信息服务管理规定的违法违规词汇
- 平台（微信）违规关键词
- 品牌自定义黑名单（例如"不能写出竞品名"）
- 儿童友好白名单（例如"血腥""暴力"在儿童设备上更严）
- 声纹通过后的成员角色（§6ter.7）：`child` 使用更严的过滤档

### 10ter.4 审核实施位置

- 技术栈：优先调用成熟服务（如阿里云内容安全 / 微信接口 / 自建关键词 + LLM 复核），**不自研敏感词表**
- v2 阶段建议双层：
  - 第一层：快速关键词过滤（毫秒级，BusinessServer 本地）
  - 第二层：成熟服务 API 语义审核（百毫秒级，异步或同步可选）
- 延迟超 500 ms 的审核不阻塞主链路：先同步第一层，第二层异步落审计；**生产正式版必须两层同步通过**

### 10ter.5 失败处理

- P1 LLM 输出被拦：TTS 改说"这个我暂时还不方便说"并记录原文到审计
- P2/P3 文本被拦：返回 `E_CONTENT_BLOCKED`；小程序侧用儿童友好文案"这句话不太合适哦，换一句试试"
- P4 生成图形被拦：返回 `E_CONTENT_BLOCKED` 或 `E_INVALID_DRAWING`；审计保留输入摘要与拒绝原因

新增错误码 `E_CONTENT_BLOCKED`，同步加入 §14 映射表。

### 10ter.6 审计

- 所有被拦截内容落 `content_audit(account_id, device_id, path, raw_hash, rule_hit, ts)`
- `raw_hash` 不落原文（原文只短暂保留用于告警复盘，30 天后硬删除）
- 敏感命中达阈值时触发运营告警（§23）

### 10ter.7 不可绕过

- 声纹即使通过也不跳过内容审核
- 主控即使发 `client.submit_write_text` 也必须过入站审核
- 没有 `skip_moderation` 参数，开发者也不能绕过
- 与 §10bis.10 精神一致

### 10ter.8 与 AI 档位的关系

- 旗舰版 LLM 可能提供"更聪明但更有风险"的回复；旗舰版**审核更严而非更松**
- Provider 切换不改变审核策略（审核是在 provider 输出之后独立做的）
- 这保证"换更好的模型"不会意外放松内容安全

## 11. 身份与设备归属

### 11.1 身份模型

- 根身份：微信 `unionid`，次级 `openid`
- 一个 `unionid` 对应一个 `account_id`（内部主键）
- JWT 载荷：`account_id + issued_at + expires_at + primary_session_id`
- 登录：`client.login(code)` → `BusinessServer` 换取 openid/unionid → 发 JWT
- 多端：同一 `account_id` 可有多个活跃 JWT，各自一个 `session_id`

### 11.2 设备身份

- 每台设备出厂写入 `device_sn`（序列号）+ `device_secret`（设备密钥）
- 激活码 `activation_code` 与 `device_sn` 一一对应，由 `BusinessServer` 在生产阶段生成
- 激活码仅首次绑定使用，绑定成功后失效
- 设备上线时向 `DeviceServer` 出示 `device_sn + device_secret`，换取一次性会话 token

### 11.3 绑定流程

```
小程序                BusinessServer                     DeviceServer         设备
  │                       │                                  │                  │
  │  login(wx code)       │                                  │                  │
  │─────────────────────►│                                  │                  │
  │◄───── JWT ────────────│                                  │                  │
  │                       │                                  │                  │
  │  bind_device(sn, code)│                                  │                  │
  │─────────────────────►│                                  │                  │
  │                       │ 校验 (sn, code) ∧ 未绑定         │                  │
  │                       │ 写入 binding(account, device)   │                  │
  │◄──── ok, device_id ───│                                  │                  │
  │                       │                                  │◄─── hello(sn,sec)┤
  │                       │                                  │─── ok, token ───►│
  │                       │◄────── device_online(sn) ────────│                  │
  │◄────── event: device_online ──────────────────────────── │                  │
```

### 11.4 归属模型与扩展位

- 当前必需：`account ──绑定── device`
- 预留 ToB 扩展：`org ──has── class ──has── account ──绑定── device`
- `BusinessServer` 表结构从第一天起把 `org_id / class_id` 作为可空外键，不阻塞 ToC，但 ToB 到来时无需迁移

### 11.5 转赠

- `client.transfer_device(target_unionid)` 发起转赠请求
- `BusinessServer` 通知目标账号，收到 `accept` 后换绑
- 转赠成功后清空该设备的历史订阅会话；历史任务保留但归档

### 11.6 出厂流程

以下流程是架构依赖项，产线必须按此执行，否则 §11.2 / §11.3 无法工作。

流程节点：

1. `BusinessServer` 提供管理端接口，由生产主管批量生成 `(device_sn, device_secret, activation_code)` 三元组，写入数据库且状态为 `unprovisioned`
2. 产线工装通过 USB 或局域网把三元组刷写到设备的 `NVS`（U8 侧）
3. 工装向 `BusinessServer` 回传"已刷写"状态，设备记录翻转为 `provisioned_unbound`
4. 激活码以卡片/二维码形式与设备一起出货
5. 终端用户首次 `bind_device(sn, activation_code)` 时，状态翻转为 `bound`，激活码作废

### 11.7 出厂数据字段

设备 NVS 存储：

- `device_sn`：可从设备管理端查询到的唯一序列号
- `device_secret`：256 位随机密钥，永不出厂后变更
- `server_host`：`DeviceServer` 接入域名，允许后续 OTA 更新
- `hw_rev`：硬件版本
- `fw_rev`：固件版本

激活码规则：

- 长度 12 位，字母+数字
- 生成时与 `device_sn` 强绑定，查询时两者必须同时出示
- 一次使用有效
- 丢失时可由管理端凭 `device_sn` 重新生成一次，旧码作废

### 11.8 不做的事

- 不支持"预绑定"：出厂时不预设账号
- 不支持"裸机抢占"：设备没有 `device_secret` 无法接入 `DeviceServer`
- 不支持"免激活码"：任何绑定必须走激活码，避免序列号遍历攻击

## 11bis. 设备联网与配网

### 11bis.1 物理唯一性

设备 U8 仅有 Wi-Fi 一种联网手段（ESP32-S3 不带蜂窝模块）。所有"上云"路径都从 Wi-Fi 出发：

```
U8 ──Wi-Fi──► 家用路由器 ──公网──► DeviceServer
```

不支持局域网直连小程序、不支持手机热点临时穿透、不支持有线。家庭无 Wi-Fi 视为不在 v2 服务范围内。

### 11bis.2 无感目标

用户视角：**点一下"添加设备"，等一会儿，添加成功**。

不要求用户：

- 输入 Wi-Fi 密码（小程序自动取家用 Wi-Fi 凭据）
- 切换手机 Wi-Fi（由小程序自动完成或绕过）
- 理解 SmartConfig / AP / BLE 等技术概念

允许例外：iOS 系统因为权限策略，可能弹一个原生确认对话框，文案由小程序写得鼓励。

### 11bis.3 双轨方案

v2 同时支持两条配网通道，小程序按环境自动选更稳的一条，用户感知一致：

| 通道 | 工作机制 | 何时优先 |
|---|---|---|
| BLE 配网 | 手机 BLE 连设备，蓝牙信道传 Wi-Fi 凭据 | 默认首选；iOS/Android 体验一致；不依赖切热点 |
| AP 配网 | 设备开 SoftAP，小程序连过去传凭据再切回 | BLE 不可用时回退；适合无 BLE 权限或老设备 |

判定顺序：

1. 小程序检测手机 BLE 是否可用，可用则走 BLE
2. BLE 不可用或 30 秒内未发现广播，回退 AP
3. AP 也失败时进入手动密码输入兜底（§11bis.6）

设备侧：U8 上电后**同时**开 BLE 广播和 AP 热点，谁先收到合法凭据就用谁，另一通道立刻关闭。

### 11bis.4 首次配网流程

```
小程序                                              U8(设备)
  │                                                    │
  │  用户点"添加设备"                                  │
  │  请求获取家用 Wi-Fi 凭据                            │
  │  (微信 getWifiList / getConnectedWifi)             │
  │                                                    │
  │  广播 BLE 扫描 + 监听设备 AP                        │
  │                                       (上电默认开)│
  │                                       BLE+AP 双开 │
  │                                                    │
  │  发现设备                                          │
  │◄──────────── BLE 广播 / AP SSID ───────────────────│
  │                                                    │
  │  通过 BLE 或 AP 发凭据                              │
  │  { wifi_ssid, wifi_pwd, server_host,               │
  │    one_time_token }                                │
  │────────────────────────────────────────────────────►│
  │                                                    │
  │                                       关闭另一通道│
  │                                       存入 NVS    │
  │                                       连家用 Wi-Fi│
  │                                                    │
  │                                       连 DeviceServer
  │                                       发 hello(sn,sec) + token
  │                                                    │
  │                                                    ▼
  │                                       DeviceServer 通知
  │                                       BusinessServer "上线"
  │◄────── 通过 §9 订阅看到 device_online ─────────────
  │                                                    │
  │  显示"添加成功"                                    │
```

`one_time_token` 由 BusinessServer 在用户点"添加设备"时签发，绑到 `(account_id, device_sn_unknown)` 临时记录上，10 分钟有效。设备拿着这个 token 上线时，DeviceServer 与 BusinessServer 联手完成 §11.3 的 binding 写入。

### 11bis.5 配网凭据存储

设备 NVS 存：

- `wifi_ssid` / `wifi_pwd`
- `server_host`（已在 §11.7）
- `wifi_provisioned_at` 时间戳，用于判断"是否已配过网"

凭据加密：

- 在 NVS 上启用 ESP-IDF 的 `nvs_encryption`
- 密钥来自 eFuse，不外泄

不存：用户账号信息。账号绑定关系存于 BusinessServer，设备只持有 `device_secret + token`。

### 11bis.6 失败兜底

可能失败：

- BLE 与 AP 都连不通：5 分钟超时后小程序提示"附近没找到设备，请检查设备指示灯"
- Wi-Fi 凭据错误：设备试连 60 秒后退回配网模式，BLE/AP 重新可见
- 路由器无法访问公网：设备能连家用 Wi-Fi 但 DeviceServer 不通，事件 `device_provision_failed_no_internet` 通过下次配网时上报，小程序友好提示
- 手动兜底：用户可在小程序"配网失败"页面**手动输入** Wi-Fi 密码，小程序通过 BLE/AP 转发。这是兜底入口，默认隐藏

### 11bis.7 重配网

家庭换路由器、换密码时需要重配网：

- 设备发现连不上原 Wi-Fi 超过 60 秒 → 自动进入配网模式（BLE+AP 同时开）
- 用户在小程序选"重新配网"时，由 BusinessServer 通过 DeviceServer 给设备发"重配网"命令；若设备已在线则下次会话生效，已离线则上次离线已自动进入配网模式
- 设备物理上提供长按 BOOT 键 5 秒强制进配网模式的物理兜底（独立于软件状态）

### 11bis.8 断网时的设备行为

设备失去外网后：

- 当前任务：根据 §12bis 处理（不自动续跑）
- 已有缓存的 segments：v2 不在设备上缓存，无离线写字能力
- 设备指示：本地 LED 表示离线状态
- 重连：每 30 秒自动重试，指数回退到上限 5 分钟
- 不主动进配网模式（避免误清凭据），除非如 §11bis.7 触发条件命中

### 11bis.9 安全约束

- BLE 配对必须使用 `Just Works` + 配网会话内 GATT 特征签名校验（防止陌生人靠近时被劫持）
- AP 必须使用 WPA2 + 设备出厂随机化的 SSID（含序列号尾 4 位）和密码
- 一次成功配网后，BLE/AP 通道立即关闭，且本次 `one_time_token` 立即作废，不允许复用
- 配网期间设备 LED 必须明显闪烁（红/蓝），便于用户判断设备状态，也便于发现异常配网

### 11bis.10 不做的事

- 不做 SmartConfig（旧 ESP-Touch）：路由器屏蔽率高、安全性差
- 不做 WPS：依赖路由器物理按钮，多数家庭不便
- 不做 Wi-Fi 多备份：只存最近一次成功的凭据，换网必走重配网，避免凭据泄漏面扩大
- 不做"邻居代配网"：账号 A 给账号 B 的设备配网这种社交分享需求 v2 不支持

## 12. 多客户端一致性

### 12.1 读 vs 写

- 读（订阅、查询）：所有绑定该设备的账号/会话都可以
- 写（下发任务、取消、暂停）：只有主控会话可以

### 12.2 主控会话

- 每台设备在 `BusinessServer` 维护一个 `primary_session_id`
- 主控默认是最近一次 `bind_device` 或 `claim_primary` 的会话
- 主控断线超过 60 秒视为放弃主控，释放给下一个 `claim` 的会话
- `claim_primary` 同账号内部优先；跨账号只在设备空闲且前主控已释放时允许

### 12.3 并发任务

- 同一设备同时只允许一个 in-flight 任务
- 新任务下发时如设备非 `IDLE`，`BusinessServer` 返回 `E_DEVICE_BUSY`
- 排队由客户端决定，`BusinessServer` v2 不实现队列

### 12.4 冲突处理

- 两个会话几乎同时 `submit`：以 `BusinessServer` 收到顺序为准，后到 `E_DEVICE_BUSY`
- 主控正在执行任务时非主控发 `cancel_task`：返回 `E_NOT_PRIMARY`
- 主控发 `cancel_task` 时设备已完成：返回 `E_TASK_ALREADY_DONE`，幂等

## 12bis. 设备掉线与任务命运

### 12bis.1 三种掉线场景

| 场景 | 检测方式 | 任务处理 |
|---|---|---|
| 设备瞬断后立即重连 | `DeviceServer` 心跳超时 < 30 秒内恢复 | 任务保留，继续等待设备端上报 |
| 设备长时掉线 | 心跳超时 ≥ 30 秒 | 活动任务置 `failed` + `E_DEVICE_OFFLINE`，设备上线后不续跑 |
| 设备端断电重启 | 设备端 `device_online` 事件且 `uptime` 回到 0 | 服务端视任务 `failed`，不做自动续跑 |

### 12bis.2 不做自动续跑的理由

- 写字机/画图任务中途断电后机械位置不可信
- 续跑需要先重新归零，但归零本身会覆盖当前墨迹位置
- 用户侧最安全的语义是"任务失败，请重新下发"
- 等未来有"断点续写"能力时再作为独立 capability，不默认打开

### 12bis.3 事件序列

掉线期间 `BusinessServer` 产生的事件：

```json
{ "event_type": "device_offline", "device_id": "dev_xxx" }
{ "event_type": "job_error", "task_id": "t_xxx", "payload": { "code": "E_DEVICE_OFFLINE" } }
```

设备重新上线后：

```json
{ "event_type": "device_online", "device_id": "dev_xxx" }
```

客户端看到 `device_online` 后可以重新下发任务。

### 12bis.4 长任务的进度保护

- `run_path` 任务若已执行过半，`BusinessServer` 在失败事件里带上 `last_progress`，便于客户端决定是重试全部还是仅重绘后半部
- v2 只提供信息，不提供自动重放；重试策略由客户端按产品形态决定

## 13. 内容资源域

### 13.1 资源类别

- 字体 `font`：中文楷/隶/行等，对应矢量字库
- 字帖 `copybook`：预制写字内容包
- 画作素材 `asset`：预制图案，仅作为 `draw_asset` fallback / demo 资源

### 13.2 资源表

- `fonts(font_id, name, version, vendor, file_ref, default_free)`
- `copybooks(copybook_id, title, font_id, text, preview, price_model)`
- `assets(asset_id, title, file_ref, preview, price_model)`
- `entitlements(account_id, resource_type, resource_id, source, expires_at)`

### 13.3 授权来源

`entitlements.source` 取值：

- `factory`：随设备出厂默认授权
- `purchase`：用户主动购买
- `gift`：赠送/活动
- `trial`：试用，有过期

### 13.4 与任务的关系

- `client.submit_write_text` 在用非默认字体时，`BusinessServer` 先查 `entitlements`
- 未授权返回 `E_NOT_ENTITLED`，小程序引导购买
- 投影缓存 `(font_id, text_hash)` 在资源吊销后不再服务（软失效）

### 13.5 资源与协议解耦

- Edge-B/C/D 上不出现 `font_id / asset_id / price_model`
- `BusinessServer` 解析完后只把 `segments + feed_rate` 发给下层
- 这保证设备协议形态与内容商业模式完全解耦，内容包改动无需改协议

### 13.6 默认免费集合

每台设备出厂时通过 `entitlements(source=factory)` 默认开通：

- 基础楷体 1 套 `font_id = kai_basic_v1`
- 基础儿童字帖包 1 套 `copybook_id = pinyin_basic_v1`
- 基础画作素材 5 件 `asset_id = starter_*`

这些条目在所有账号都可用，不依赖账号付费。新设备绑定后由 `BusinessServer` 自动注入。

### 13.7 缓存失效判定时机

投影缓存键 `(font_id, text_hash)` 或 `(asset_id, params_hash)`，在以下时机失效：

- 命中后：先查 `entitlements` 是否仍有效，无效则视为未命中
- 资源版本升级 `font.version` 改变：`font_id` 变更，旧缓存自然失效
- 资源被运营下架：`fonts.status = revoked`，`BusinessServer` 拒绝命中
- 试用过期：`expires_at < now`，命中视为未命中

判定一律在缓存读取时进行，不依赖被动清理任务。这避免"缓存还在 → 用户已失权 → 仍能用"的越权窗口。

### 13.8 字体作者与版权

- `fonts` 表的 `vendor` 字段记录字体提供方，用于版权追溯
- 商用字体必须有授权凭证存档，存档不在 `BusinessServer` 数据库内（不属于运行时数据）
- 用户上传自定义字体不在 v2 范围内，由 v3 决定

### 13.9 AI 能力作为资源域

LLM / ASR / TTS / Vision 纳入 §13 资源域，与字体/字帖/素材同级。理由：

- AI 的换新、成本、授权规则与内容包同构
- 用户应该买到的是"更聪明的小智"，不是"某个技术栈"
- 商业侧的成本敏感点（LLM 调用费）必须有明面的计费位

对用户可见的是**能力档**，对运营可见的是**底层 provider 池**。

### 13.10 AI 资源表

`ai_providers(provider_id, category, vendor, model_name, api_endpoint, price_unit, quality_tier, status)`

- `category`：`llm / asr / tts / vision`
- `vendor`：`deepseek / openai / anthropic / volcengine / aliyun / local / ...`
- `quality_tier`：`basic / advanced / flagship`
- `status`：`active / deprecated`，deprecated 的仍可用到现有订阅到期

`ai_plans(plan_id, name, llm_provider_id, asr_provider_id, tts_provider_id, vision_enabled, quota_monthly, price_model)`

- 每个 plan 是一组 provider 绑定
- v2 阶段定义 3 档官方 plan：

| plan_id | 中文名 | LLM | ASR | TTS | Vision |
|---|---|---|---|---|---|
| `plan_basic` | 基础 | 国产通用 LLM | 通用 ASR | 通用 TTS | 关 |
| `plan_advanced` | 进阶 | DeepSeek / 同档 | 定制 ASR | 音色可选 | 关 |
| `plan_flagship` | 旗舰 | DeepSeek 多模态 / 同档 | 定制 ASR | 音色可选 | 开 |

`entitlements` 复用 §13.2 已有的结构，`resource_type='ai_plan'`。

### 13.11 能力档的用户可见面

小程序里的呈现规则：

- 呈现"能力档"而不是模型名：`基础 / 进阶 / 旗舰`
- 附简短说明："进阶版可以听懂更复杂的要求，能画你说的图"
- **不暴露**具体 provider/vendor/model name
- 小程序不提供"强制切换到 GPT-4o"等选项
- 账号设置里可以看"当前档位 + 到期时间"，没有模型开关

设备级设置（如果后续需要）在 `device_caps.ai_plan_override` 中表达，但 v2 暂不开放。

### 13.12 切换与生效时机

- 能力档变更在**下一次语音会话开始时**生效，不做会话中热切换
- 原因：模型热切换时序复杂，用户语音不同步会产生"怎么回答怪怪的"异常体验
- 会话开始时 DeviceServer 向 BusinessServer 拉取当前 `ai_plan`，缓存到会话内
- 会话结束或超时 5 分钟后重新拉取

### 13.13 失效与降级

- `provider.status=deprecated`：现有订阅继续用到期，新订阅不可选
- Provider 服务故障：DeviceServer 自动降级到同 category 内 `quality_tier` 较低的 provider；降级事件通过 §23 监控告警（见后文）
- 账号订阅过期：自动回落到 `plan_basic`
- 所有自动降级必须落 `safety_audit` 或类似审计表（见 §23）

### 13.14 AI 使用与内容审核的关系

LLM 的任何输出都要过 §10ter 内容审核，不允许以"底层模型已经做了审核"为由跳过（见下文 §10ter）。

## 14. U1 状态机与错误码

U1 层严格沿用 v1 §8 / §9 / §10 / §11，v2 不改。

为便于对客户端呈现，v2 增加一层"业务可读错误码"映射：

| 客户端错误码 | 含义 | 对应 U1 错误码 / 情形 |
|---|---|---|
| `E_DEVICE_BUSY` | 设备非空闲 | v1 `E004 BUSY` |
| `E_DEVICE_OFFLINE` | 设备掉线 | 心跳超时 |
| `E_NOT_HOMED` | 未归零 | v1 `E001 NOT_HOMED` |
| `E_OUT_OF_RANGE` | 超工作区 | v1 `E002 OUT_OF_RANGE` |
| `E_INVALID_PARAM` | 参数非法 | v1 `E003 INVALID_PARAM` |
| `E_LIMIT_TRIGGERED` | 限位告警 | v1 `E005 LIMIT_TRIGGERED` |
| `E_HOMING_FAILED` | 归零失败 | v1 `E006 HOMING_FAILED` |
| `E_ABORTED` | 受控中止 | v1 `E007 MOTION_ABORTED` |
| `E_ESTOP` | 急停 | v1 `E008 ESTOP_TRIGGERED` |
| `E_TIMEOUT` | 执行超时 | v1 `E011 TIMEOUT` |
| `E_INTERNAL` | 其他内部错误 | v1 `E009/E010/E012` |
| `E_AUTH` | 未登录/JWT 失效 | Edge-A 鉴权 |
| `E_FORBIDDEN` | 越权 | Edge-A 鉴权 |
| `E_NOT_FOUND` | 资源不存在 | 业务侧 |

| `E_NOT_PRIMARY` | 非主控 | 业务侧 |
| `E_TASK_ALREADY_DONE` | 任务已完成/幂等 | 业务侧 |
| `E_CONTENT_BLOCKED` | 内容审核拦截 | §10ter |
| `E_DEVICE_UPDATING` | 设备升级中 | §21.5 |
| `E_CLIENT_UPGRADE_REQUIRED` | 客户端版本过低 | §19.6 |
| `E_FW_UPGRADE_REQUIRED` | 设备固件过低 | §19.6 |
| `E_NO_PAPER` | 纸张未就绪 | §26.5 |

原则：

- 设备侧错误码对客户端永远隐藏在 `E_*` 之后
- 必要时在 `payload.detail` 里带设备码，便于排障但不展示
- 新增错误码必须先在此表登记，不允许在代码里偷加

## 14bis. 故障恢复路径

错误码只是描述"出了什么"，本节定义"出了之后怎么走回 IDLE"。

### 14bis.1 ALARM 恢复

触发：`E_LIMIT_TRIGGERED / E_HOMING_FAILED / E_OUT_OF_RANGE`（运行时）

恢复路径：

1. U1 进入 `ALARM`，所有运动任务被丢弃
2. `BusinessServer` 当前活动任务置 `failed`，事件含 `code=E_LIMIT_TRIGGERED` 等
3. 客户端收到事件后建议 UI 引导：检查机械、移除阻挡
4. 客户端调 `client.submit_motion(clear_alarm)`（v2 暂归类到 `client.submit_motion` 内部子动作，命名延后）
5. `BusinessServer` 把请求转换为 U1 的 `clear_alarm` 控制（v2 §18 中提示 v1 §10"故障清除并显式复位"已经留位）
6. U1 回 IDLE，事件 `device_status: IDLE`

### 14bis.2 ESTOP 恢复

触发：`E_ESTOP`，U1 BOOT 按键或软件指令

恢复路径：

1. U1 进入 `ESTOP`，最高优先级中断
2. `BusinessServer` 任务置 `failed: E_ESTOP`
3. 用户按 §10bis.12 排查导致急停的原因并处理（例如掉物件、限位压住）；若是整机硬切断，需先恢复电源
4. 客户端调 `client.submit_motion(reset_estop)` 通知 `BusinessServer`，`BusinessServer` 通过 Edge-B → DeviceServer → U8 → U1 发 reset
5. U1 在确认当前无异常的前提下回 IDLE
6. 客户端建议 UI 强制走"复位 → 归零"两步，归零成功后才允许下发新任务

### 14bis.3 ERROR 恢复

触发：`E_INVALID_PARAM / E_INTERNAL`（参数/协议类）

恢复路径：

- 客户端不需要做特殊解锁动作
- 排错责任在调用方，重新提交即可
- `E_INTERNAL` 累积超过阈值时 `BusinessServer` 在 `device_status` 里加 `health=degraded` 提示

### 14bis.4 设备掉线恢复

见 §12bis。

### 14bis.5 任务取消的恢复

`STOP` 是受控中止，不算故障：

- U1 把当前任务安全停止后回到 IDLE
- 任务状态 `cancelled` + `code=E_ABORTED`
- 客户端可立即下发下一个任务，不需要复位流程

附加约束：

- 若底层控制器无法把某次 `STOP` 实现为受控停止，而是落成 reset、失步、不可恢复中断或其他异常结果，则该次操作不得继续标记为 `cancelled`
- 这类情况必须上升为 `failed`，并返回对应错误码；只有真正受控结束的停止才允许映射为 `E_ABORTED`

### 14bis.6 一致性约束

- 任何恢复路径都必须以"设备明确发出新一轮 `device_status: IDLE`"为终点，UI 才允许进入"可下任务"状态
- 客户端不允许在事件流里看到"猜测的 IDLE"就放行新任务

## 15. 协议字段表扩展

### 15.1 Edge-A（Client ↔ BusinessServer）

- HTTPS 请求公共头：`Authorization: Bearer <jwt>`

```

上行事件：

```json

  "event_id": "uuid",
  "device_id": "dev_xxx",

- `STOP` 命令表示"请求受控停止当前任务"，不是 reset 的别名

## 19. 版本演进与推送门禁

### 19.1 版本规则

- 架构定稿版本号：主号+次号，例如 v2 / v2.1
- 主号升级：协议边界、角色数、能力分组发生变化
- 次号升级：已有边界内字段增减或错误码扩充
- 次号内向后兼容，主号升级允许不兼容但必须提供迁移说明

### 19.2 推送门禁（v1 §22.5 扩展）

任何推送到主分支前，必须满足：

- 不存在协议边界违规（例如 Client 直连 DeviceServer、BusinessServer 生成 G-code）
- 新增错误码已写入 §14 映射表
- 新增 capability 已在 §7/§8 登记
- 文档与实现一致，或已在 §18 差距表登记差异

### 19.3 v2 之后可能的版本

- v2.x：增加客户端 capability 或事件字段
- v3：纳入 ToB 多租户或 AI 内容生成管线时再启，主号升级

### 19.4 版本协商与 forward-compat

OTA 灰度期、小程序审核延迟期、客户端缓存期都会出现"多版本并存"。各边界协商规则：

**Edge-A（小程序 ↔ BusinessServer）**

- 每次 HTTPS 请求带 `X-Client-Ver` 头；WSS 首帧 `auth` 带 `client_ver`
- BusinessServer 读取后决定返回字段集合；无法兼容时返回 `E_CLIENT_UPGRADE_REQUIRED`，小程序显示"请更新到最新版"
- 小程序侧在收到**未知字段**时静默忽略（forward-compat 原则）
- 小程序侧发送**未知字段**时 BusinessServer 静默忽略（不报错，便于客户端先行）

**Edge-B（BusinessServer ↔ DeviceServer）**

- 内部服务协议版本号 `X-Internal-API-Ver` 在请求头里
- server-to-server 升级要求先 BusinessServer 后 DeviceServer，或反之，明确先后与降级路径
- 任何不兼容变更先发 dev 环境验证一周

**Edge-C（DeviceServer ↔ U8）**

- WSS `hello` 帧必须携带 `fw_ver` 与 `proto_ver`
- DeviceServer 按 `proto_ver` 启用不同编解码器
- 不兼容时 DeviceServer 可拒绝连接并触发强制 OTA（见 §21）

**Edge-D（U8 ↔ U1）**

- 每个 `@{json}` 帧可携带 `proto_ver` 可选字段，缺省按 v1 处理
- U1 在拒识别到更高版本时返回 `E_INTERNAL` 并带 `supported_proto_ver`
- U1 固件升级与 U8 固件升级可能错位（§21.7 透传存在延迟），此时允许 U8 降级编码保底

### 19.5 字段演进规则

- 增字段：任何一边都可加，对方静默忽略
- 改字段类型：主号升级
- 减字段：次号可减，但需要提前 2 个次号版本标记 `deprecated` 再实际删除
- 改字段含义：主号升级，不允许同名字段变语义
- 错误码：可加不可改含义；`E_*` 值变更等于主号升级

### 19.6 强制升级

在以下情况 BusinessServer 可以强制拒绝客户端或设备：

- 客户端版本低于 `min_client_ver`（安全修复等紧急原因）
- 设备固件版本低于 `min_fw_ver`（同上）
- 强制拒绝时返回 `E_CLIENT_UPGRADE_REQUIRED` / `E_FW_UPGRADE_REQUIRED`，并带升级指引

新增错误码加入 §14 映射表。

### 19.7 多版本同时在线的约束

- `task_id` 格式跨版本不变
- `seq` 单调递增语义不变
- 事件订阅在版本切换时不可中断，必须按 §9.3 断线补齐

## 21. OTA 固件升级

设备出货后 U8 / U1 固件都会有 bug、改进、协议演进。v2 必须定义 OTA 能力，否则设备变"一次性产品"。

### 21.1 范围

- U8 固件：主要升级对象，频率最高
- U1 固件：频率低，但必须可升；通过 U8 透传（Edge-D 传包）
- `BusinessServer` / `DeviceServer`：不是 OTA 范围，走常规 server 部署

### 21.2 固件包管理

BusinessServer 表 `firmware_releases(release_id, target, version, channel, url, sha256, min_prev_version, released_at)`：

- `target`：`u8 / u1`
- `channel`：`dev / beta / stable`
- `min_prev_version`：最低起跳版本，低于此版本必须先跳到中间版本
- `url + sha256`：下载地址与校验
- 包文件存 OSS 或同等对象存储，CDN 下载

### 21.3 灰度发布

- 发布流程：`dev → beta → stable`
- 每档按账号 ID hash 或设备 ID hash 分批（10% → 50% → 100%）
- 灰度期间 BusinessServer 监控升级失败率、任务失败率，异常超阈值自动停灰并告警（§23）
- Stable 发出后至少 24 小时观察期才能发下一版

### 21.4 设备端流程

```
设备上线 ──► DeviceServer 通知"有新版 vX.Y"
                │
                ▼
        U8 判断：当前是否在执行任务？
                │
        空闲 ◄─┘└─► 忙，延后到下次空闲
                │
                ▼
        U8 下载 → 校验 sha256 → 双分区写入 → 重启
                │
        失败 ◄─┘└─► 成功：上线后上报 `fw_update_done`
                │
                ▼
        回滚到 A/B 另一分区 → 上报 `fw_update_failed`
```

### 21.5 任务中禁止升级

- 升级必须在 `IDLE` 状态开始
- 已开始升级期间 DeviceServer 拒绝新任务下发，返回 `E_DEVICE_UPDATING`
- 用户语音触发任务也一并拒绝，并由 TTS 友好告知"正在升级，一会儿再来"
- 新增错误码 `E_DEVICE_UPDATING`，同步加入 §14 映射表

### 21.6 降级与回滚

- ESP32-S3 使用双分区 (A/B slot) 即 ESP-IDF `app_rollback_is_possible`
- 升级后首次开机成功连上 DeviceServer 并跑完自检（§24）才标记 `valid`
- 未 `valid` 情况下再次启动视为失败，自动回滚到上一分区
- 用户侧不可见，小程序不显示"回滚"操作
- 运营侧可在管理后台强制下发"回滚到 vX.Y"指令，紧急时使用

### 21.7 U1 固件透传

- U1 不联网，升级路径为：`DeviceServer → U8 → UART → U1`
- U8 作为 U1 bootloader 桥：收到 `u1_firmware` 目标时，U8 进入"U1 烧录模式"（禁用运动命令、拉 U1 BOOT/RESET、通过 UART 传包、校验）
- U1 固件大小 < 2 MB（Grbl_Esp32 体量可控）
- 失败时 U1 仍能回到出厂固件（Grbl 有自己的 OTA 分区机制，复用或暂不启用由实施阶段定）

### 21.8 安全

- 所有固件包必须签名：`BusinessServer` 用私钥签，设备用出厂预置公钥验
- 包未过签名校验拒绝升级，事件 `fw_signature_invalid`
- 公钥可通过后续 OTA 升级滚动，但第一次公钥必须在出厂时烧入

### 21.9 用户可见面

- 小程序"设备详情"页显示当前固件版本、上次升级时间
- 有新版时小程序提示"设备将在下次空闲时自动升级"
- **不提供**用户手动触发升级按钮（避免用户误操作打断任务）
- **不提供**用户选择具体版本（灰度由 server 决定）
- 升级中设备 LED 特殊闪烁（橙色/紫色，区别于配网/在线三态）

### 21.10 不做的事

- 不做 P2P 升级（邻居设备互相分发）
- 不做局域网直下升级（所有升级走公网）
- 不做静默降级（降级必须在管理后台显式触发）

## 22. 上线合规清单

中国市场儿童硬件 + 微信小程序 + 公网 server 的合规要求集中且严格。**没满足这些就上不了线，更上不了规模**。

### 22.1 微信小程序提审要点

- 类目：选 "智能硬件 / 智能家居" 或 "工具 / 智能硬件"
- 资质：需要营业执照，类目可能需要 ICP 增值电信业务经营许可证
- 隐私协议：必须有显著入口，包含数据收集种类、用途、保留期、共享方
- 用户授权：麦克风 / 蓝牙 / Wi-Fi / 相册（如果有）必须**单独授权**，不能默认勾选
- 涉及未成年人：在协议中显著标注"本产品面向 3~12 岁儿童使用"，并补充家长监护说明

### 22.2 数据合规

| 法规 | 范围 | 关键约束 |
|---|---|---|
| 个人信息保护法（PIPL） | 所有用户数据 | 最小必要、单独同意敏感数据、可删除 |
| 数据安全法 | 数据流转 | 重要数据本地化存储 |
| 网络安全法 | 联网产品 | 等保备案 |
| 未成年人保护法 | 儿童数据 | 14 岁以下监护人同意 |
| 数据出境管理办法 | 海外 LLM | 评估 / 申报 / 不出境 |

声纹（§6ter）、儿童语音、家长账号都是敏感个人信息。设计时遵循：

- 默认最小化采集
- 单独同意（注册声纹时弹独立确认）
- 14 岁以下账号绑定家长 unionid，下发任务/查看记录均需家长授权
- 提供"删除账号"入口，删除后 30 天内硬删除全部业务数据

### 22.3 数据出境

- 默认所有数据**不出境**：服务器全部部署在中国境内
- LLM/ASR/TTS provider 优先选择国内：DeepSeek / 智谱 / 通义 / 火山 / 阿里云 / 腾讯云
- 选用海外 provider（OpenAI / Anthropic / Google）时必须：
  - 走个人信息保护影响评估（PIPIA）
  - 用户在协议中**单独同意**数据出境
  - 评估通过后向网信办申报或备案（看数据量级）
- v2 阶段建议**只用国内 provider**，避免出境合规复杂度，等业务规模再评估

### 22.4 ICP 备案与等保

- 公网域名（DeviceServer / BusinessServer 对外暴露的）必须 ICP 备案
- 业务用户量超过阈值后需做网络安全等级保护备案（建议 v2 上线前预备 2.0 二级）
- 备案主体与营业执照一致

### 22.5 内容生成合规

- 调用国产生成式 AI 服务的必须遵循《生成式人工智能服务管理暂行办法》
- 生成式 AI 服务应在备案目录内
- 内容审核日志保留 6 个月以上
- §10ter `content_audit` 表保留期从严，至少 180 天

### 22.6 硬件合规

- 3C 认证：含电源 / 电机 / 激光 / 蓝牙 Wi-Fi 的硬件必须做
- 无线电发射设备型号核准（SRRC）：Wi-Fi / 蓝牙模块通常已带，需复用证书
- 锂电池（如果有）：UN38.3
- 玩具类合规（GB 6675）：如果走玩具类目，强制项

### 22.7 儿童产品合规

- 内容审核档位为 `child` 时使用更严白名单（§10ter.3）
- 不开放内置社交分享（朋友圈分享需家长操作）
- 儿童语音不上传至广告体系
- 设备外观与说明书必须有儿童使用警示

### 22.8 隐私政策模板要点

- 必须列清楚：
  - 收集了哪些数据（账号、声纹、设备、任务、语音、视觉）
  - 用途（提供服务、内容审核、安全审计）
  - 保留期（绑定关系永久、声纹至解绑、语音不留存、审计 90 天）
  - 共享方（OTA 包 CDN、第三方 LLM/ASR/TTS、内容审核服务）
  - 用户权利（查询、更正、删除、拒绝处理）
- 隐私政策版本变更时必须重新征得同意
- 14 岁以下用户的隐私政策需有家长签署版本

### 22.9 不做的事

- 不在 v2 阶段做出口（海外发售涉及 GDPR / COPPA / CE / FCC，独立项目）
- 不做用户互相分享设备（社交分享带来更多合规面）
- 不做内置内容生成的图片/视频分享（防止违规内容外溢）

### 22.10 实施责任

- 合规清单的具体落实由产品/运营/法务负责
- 架构层只承诺：技术实现不阻塞合规、关键数据点都有审计位、敏感数据有删除接口
- v2 §16.2 BusinessServer 实施清单中已包含：声纹删除、内容审计、账号注销基础设施

## 23. 日志、监控、告警

`safety_audit` / `content_audit` 是业务审计，保留的是"出了事查得到"。本节定义运行时**主动发现问题**的能力。

### 23.1 结构化日志

所有角色统一 JSON 结构日志：

```json
{
  "ts": "2026-01-01T10:00:00Z",
  "level": "INFO|WARN|ERROR",
  "role": "client|biz|dev|u8|u1",
  "trace_id": "uuid",
  "task_id": "uuid?",
  "account_id": "?",
  "device_id": "?",
  "msg": "short description",
  "payload": {}
}
```

### 23.2 trace_id 传递

贯穿五角色的 trace_id 规则：

- 由 Client 发起时在 Edge-A 首包生成
- 由语音发起时在 DeviceServer 接收语音帧时生成
- BusinessServer 在 §6bis.3 / §6.1 落库时把 `trace_id` 写进 `tasks` 表
- Edge-B/C 消息头透传 trace_id
- U8 把 trace_id 压缩成短串（16 字节）通过 UART 传给 U1；U1 日志打短串
- 日志搜索时按 trace_id 拼全链路

### 23.3 核心指标

BusinessServer 指标（Prometheus 风格）：

- `biz.task.submitted`（by capability）
- `biz.task.rejected`（by error code）
- `biz.task.done` / `biz.task.failed`
- `biz.event.lag_seconds`（事件从 DeviceServer 到客户端的延迟 p50/p95/p99）
- `biz.subscription.active`（活跃订阅数）
- `biz.content_audit.blocked`（按 path）
- `biz.ai_provider.degrade`（自动降级次数）

DeviceServer 指标：

- `dev.session.active` / `session.churn`
- `dev.asr.latency_ms` / `dev.tts.latency_ms` / `dev.llm.latency_ms`
- `dev.voiceprint.match` / `voiceprint.miss` / `voiceprint.strangers`
- `dev.device.online` / `device.offline`

U8 指标（通过 DeviceServer 上报）：

- `u8.wifi.rssi` / `u8.wifi.reconnect`
- `u8.ota.updating` / `u8.ota.failed`
- `u8.uart.u1.timeout`

### 23.4 关键告警

- BusinessServer / DeviceServer 进程不可用：立即告警
- `biz.task.failed` 比例 > 5% 持续 5 分钟：告警
- `dev.device.offline` 激增：告警
- OTA 灰度期失败率 > 3%：自动停灰并告警（§21.3）
- `biz.content_audit.blocked` 激增：告警运营
- AI provider 故障降级：告警运营
- Wi-Fi 配网失败率异常：告警产研

告警通道：企业微信 / 钉钉 / 邮件；严重级别走电话

### 23.5 日志保留

- 生产环境日志保留 30 天（可检索）
- 审计类数据按 §10bis.11 / §10ter.6 / §22.5 各自约束
- 日志中不打印敏感字段：`device_secret` / 声纹向量 / 原始语音 / JWT 全值 / 用户真实姓名

### 23.6 日志脱敏

- `account_id` 打 hash 后 8 位
- `device_sn` 打 hash 后 8 位
- `openid / unionid` 永不入日志
- 手机号 / 邮箱（如有）仅脱敏形式

### 23.7 埋点与用户行为分析

- 业务埋点（留存、转化）与技术日志分离系统
- 小程序埋点遵循 §22 合规要求，儿童数据不入广告体系
- 埋点事件字段与接口字段严格分离，避免"日志里意外多了一列用户画像"

### 23.8 不做的事

- 不在生产环境打印完整原始请求/响应 body
- 不做跨租户日志汇总视图（v2 只有一个主租户，v3 视情况再说）
- 不把日志外发给设备厂商代管（数据安全法要求）

## 24. 设备自检与诊断

机械设备一定会坏，用户不会修，客服需要线索。本节定义让普通家长也能操作、让客服能拿到信息的自检体系。

### 24.1 开机自检

U8 上电后依次自检：

1. NVS 凭据完整性
2. Wi-Fi 配网状态（已配 / 未配）
3. 与 U1 的 UART 连通性（发 `GET_STATUS` 并期望响应）
4. U1 返回的 `homed / limit / sensor` 状态
5. 麦克风/扬声器回环（轻声测试音）
6. 摄像头（如果 `ai_plan.vision_enabled`）

自检结果：

- 全通过：正常进入运行
- 配网未完成：进配网模式（§11bis）
- 关键项失败：LED 特殊闪烁（红色长亮）+ 通过日志上报 BusinessServer

### 24.2 自检不做的事

- 不做破坏性自检（不动电机到极限）
- 不做会留痕迹的自检（不落笔、不开激光）
- 不需要用户配合（纸/笔不影响自检）

### 24.3 家长触发诊断

小程序"设备"页有"健康检查"按钮。触发后：

1. BusinessServer 下发 `capability=diagnostic`
2. U1 按预设脚本动作：空行程扫一个小方框、抬/落笔测试、舵机归中
3. U1 上报：行程偏差 / 步进响应 / 限位触发次数 / 传感器读数
4. BusinessServer 生成诊断报告
5. 小程序显示健康评分 + 异常提示

诊断涉及真实运动，需要确认："请先放好纸张但不要装笔"。

### 24.4 诊断报告字段

`diagnostic_reports(report_id, device_id, account_id, ts, fw_ver, result_json)`

`result_json` 含：

- 各轴步进响应率
- 限位触发次数
- HX711 压力基线（有压力传感器时）
- 舵机归位误差
- Wi-Fi RSSI
- 最近 24h 任务失败率
- 自检失败项

报告保留 90 天。用户可在小程序主动查看最近 3 份。

### 24.5 客服查看

- 客服后台按 `device_sn` 查最近诊断报告
- 用户在小程序可选"授权客服查看"，授权有效期 72 小时
- 未授权时客服仅看脱敏摘要

### 24.6 诊断失败处理

- 单次失败：提示用户检查机械部分
- 连续 3 次失败：建议联系客服，并在小程序发一键"提交工单"按钮
- 工单提交时自动打包最近日志（已脱敏）+ 最近诊断报告

### 24.7 远程诊断不等于远程控制

- 客服即使拿到授权也**不能**下发运动任务
- 客服只能读诊断报告
- 若需要远程控制，必须用户在小程序显式点"允许远程协助"，并落到 §10bis.11 审计
- 远程协助会话结束后立即失效

## 25. 售后与 RMA

设备会寄修、退货、转卖。绑定关系必须有干净的结束态。

### 25.1 设备状态机

`devices.status` 状态集：

- `unprovisioned`：已生成激活码，未刷写
- `provisioned_unbound`：已刷写出厂凭据，未被用户绑定
- `bound`：正常使用中
- `rma_in_progress`：返厂维修中
- `returned`：已退货，设备冷冻
- `disposed`：报废

### 25.2 状态转移

- `unprovisioned → provisioned_unbound`：产线刷写完成
- `provisioned_unbound → bound`：用户首次激活绑定
- `bound → rma_in_progress`：客服受理报修，暂时解除绑定但保留账号关联
- `rma_in_progress → bound`：维修完成重上线；若需要换机，激活码重发
- `bound → returned`：用户退货确认
- `returned → provisioned_unbound`：清洗凭据、生成新激活码后重新上架；原账号的任务/资源保留在账号上
- 任意状态 → `disposed`：设备丢失或硬损毁

### 25.3 转赠（§11.5）与 RMA 的区别

- 转赠：设备仍归原账号体系，转移到新 unionid
- RMA：设备暂时或永久离开账号体系

### 25.4 退货后的数据处理

退货流程中：

- 绑定关系在 `returned` 状态时解除
- 设备上的 `wifi_ssid / wifi_pwd` 在刷机回厂时清空（产线负责）
- 账号侧的任务/语音/诊断报告数据保留，与账号绑定，不随设备走
- 再上架时所有凭据全新生成

### 25.5 坏件

- `disposed` 后激活码永久作废
- 若客户称设备遗失，设备上一次成功通信超过 90 天即视为遗失，客服可标 `disposed`
- 一旦 `disposed`，即使设备意外上线也拒绝会话

### 25.6 数据归档

- 账号注销时（§22.2）与设备状态无关，注销操作走账号注销流程
- 设备 `disposed` 并不自动删除账号数据
- 账号注销会解除所有设备绑定关系，设备自动回落 `provisioned_unbound`

## 26. 耗材与笔纸状态

### 26.1 硬件现状

当前 PCB 无笔/纸/墨水传感器：

- 没有笔压传感器（HX711 用于称重，不用于笔状态）
- 没有墨水余量检测
- 没有纸张在位检测

v2 阶段不依赖传感器，**以用户手动标记为主**。

### 26.2 耗材字段

`device_supplies(device_id, pen_installed_at, pen_ink_percent_est, paper_slot_state, updated_at)`

- `pen_installed_at`：用户标记的最近一次换笔时间
- `pen_ink_percent_est`：估算墨水（按累计里程估算，不是实测）
- `paper_slot_state`：`empty / loaded / unknown`，由用户标记

### 26.3 用户操作

小程序"耗材"页：

- "我换了新笔"：复位 `pen_installed_at` 与里程
- "纸张已放好"：设为 `loaded`
- "没纸了"：设为 `empty`
- 任务完成自动提示"是否换纸"

### 26.4 里程估算

- BusinessServer 每次 `run_path` 完成后累加该任务的路径长度到 `pen_mileage_mm`
- 超过经验阈值（例如 5000 mm）后小程序提示"笔可能快写不出了"
- 用户换笔后复位

### 26.5 安全联动

- 用户标记"纸张未就绪"时，BusinessServer **拒绝** `write_text / draw` 提交，返回 `E_NO_PAPER`
- 这是"软联动"，防止白板上硬写
- 新增错误码 `E_NO_PAPER`，加入 §14 映射表

### 26.6 硬件迭代预留

未来硬件加入纸张传感器 / 笔压传感器：

- 新增 U1 `get_supplies_state` 私有协议命令
- `device_supplies` 增加 `paper_detected`、`pen_pressure_valid` 字段
- 用户标记字段改为"备选真相"，实测优先
- 不需要改动 Edge-A 接口结构

## 20. 一句话总结

v2 在 v1 的设备-协议闭环之外，补齐了客户端、业务分层和商业化约束：

- 用户目标由 `Client` 表达
- 业务合法性由 `BusinessServer` 裁决并持久化
- 设备通道由 `DeviceServer` 维持
- 翻译由 `U8` 承担
- 安全执行由 `U1` 守底

商业模式的任何变化只影响 `BusinessServer` 与 `Client`，不会外溢到设备协议。
