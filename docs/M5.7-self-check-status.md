# M5.7 startup self-check status

Date: 2026-05-15

Scope implemented:
- U8 emits a `type:"self_check"` startup event after activation, OTA/version checks, and protocol initialization.
- The startup event includes `device_id`, `check_id:"startup"`, `scope:"startup"`, `status`, and per-check results for `nvs`, `wifi`, `u1_uart`, and `audio`.
- The P1 board checks U1 UART by issuing a short `GET_STATUS` over the existing private UART protocol.
- xiaozhi-server forwards `type=self_check` to manager-api at `/internal/v1/self_check`.
- manager-api accepts the self-check event and broadcasts an Edge-A `event_type:"self_check"` event on the device topic.

Design-time limits:
- Audio check verifies codec readiness and sample-rate configuration, not acoustic loopback.
- Wi-Fi check verifies the active network object after connection, not RSSI or internet reachability.
- Later M5.8 work adds local diagnostic persistence in `device_self_check_events` and exposes the latest five diagnostics through `POST /api/v1/devices/{deviceId}/self-check/history`.
- No real hardware boot cycle was performed.

## Startup Self-Check Evidence Gap Record

Use this structure when startup self-check evidence is missing for a release candidate. It is not evidence of real boot-cycle diagnostic coverage.

- missing evidence scope: real hardware boot cycle, U8 startup `self_check` emission, U1 UART `GET_STATUS` response, codec readiness, Wi-Fi active-network validation, xiaozhi-server forwarding, manager-api ingestion, Edge-A broadcast, or persisted `device_self_check_events` history visibility
- environment: staging U8 device, U1 mechanism, release-candidate firmware, xiaozhi-server deployment, BusinessServer deployment, bound mobile account, and device id
- owner: named firmware, backend, or mobile owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep startup self-check labeled design-time, keep manual health-check entry visible, and block release if boot diagnostics are absent or cannot reach mobile history
- risk acceptance: firmware owner, backend owner, mobile owner, operations owner, and release manager approval reference
- rollback trigger: startup event is not emitted after boot, any required check is missing, U1 UART check hangs or reports unsafe state, xiaozhi-server forwarding fails, manager-api ingestion fails, or mobile cannot load the latest diagnostic history
- follow-up evidence: hardware boot serial log, `self_check` payload, U1 `GET_STATUS` transcript, codec/Wi-Fi check output, xiaozhi-server forwarding trace, manager-api ingest log, Edge-A event, and mobile history screenshot
