# M6.5 Device Transfer Status

Date: 2026-05-16

## Evidence Boundary

This status is local/design-time evidence only. It is not a completion declaration and does not replace platform push delivery, native tabBar screenshots on real WeChat clients, background reminder delivery, or approved release deferral evidence.

## Scope

- Current account can request a device transfer to a target `unionid`.
- Target account can accept a pending transfer.
- Source account can cancel its own pending transfer before it is accepted.
- Accepting a transfer moves the active binding to the target account.
- Historical tasks stay on the original account because `tasks.account_id` is not migrated.
- Successful transfer clears old members and voiceprints for that device and asks DeviceServer to clear active local voiceprint cache.
- Target account can list transfers waiting for confirmation.

## Implementation

- `device_transfer_requests` stores `pending|accepted|cancelled` transfer requests.
- `DeviceTransferService` provides:
  - `requestTransfer(deviceId, targetUnionid)`
  - `acceptTransfer(transferId)`
  - `cancelTransfer(transferId)`
  - `listPendingIncomingTransfers()`
- API endpoints:
  - `POST /api/v1/devices/{deviceId}/transfer`
  - `POST /api/v1/device-transfers/{transferId}/accept`
  - `POST /api/v1/device-transfers/{transferId}/cancel`
  - `POST /api/v1/device-transfers/pending-incoming`
- `DeviceServerMotionGateway.clearVoiceprintCache` calls DeviceServer:
  - `POST /internal/v1/voiceprints/cache/clear`
- transfer request and acceptance binding ownership checks select active bindings by `updated_at` desc and binding `id` desc before moving ownership.
- Authenticated transfer entrypoints reject an existing `status = deleted` account before binding lookup or transfer mutation, so old tokens from a deleted account cannot create or act on device transfer requests.
- Voiceprint enrollment also selects account/device active bindings by `updated_at` desc and binding `id` desc before creating member and voiceprint rows.
- xiaozhi-server provides `VoiceprintCacheHandler` and `ActiveVoiceprintCache.clear_device`.
- BusinessServer voiceprint cache export first resolves the latest active binding by `updated_at` desc and binding `id` desc, then returns only active voiceprints for that binding account and device. If no active binding exists, the exported cache is empty.
- manager-mobile device detail page supports:
  - request transfer by target `unionid`
  - cancel pending transfer by `transferId`
  - accept pending transfer by `transferId`
- manager-mobile device list page supports:
  - listing pending incoming transfers for the current account
  - accepting a pending incoming transfer directly from the list
  - showing an in-app pending incoming transfer count badge
  - updating a best-effort native tabBar badge on the home tab after pending transfer refreshes
  - reapplying the stored local native tabBar badge count when the app returns to `onShow`
- Product notification runbook `ops/runbooks/m6-product-notifications.md` defines the future platform push/background reminder contract and acceptance drill for pending incoming transfers.
- `ProductNotificationOutboxService` writes a `pending_device_transfer` row to `product_notification_events` for the target account when a transfer request is created. The row stores only safe semantic fields and a deep link, not activation codes or account secrets.
- The local outbox enqueue API for device transfers receives only recipient, device, and transfer references; it does not accept activation codes, account secrets, or other binding payload fields.
- `ProductNotificationOutboxService.buildSafeProviderPayload(...)` exposes only `event`, `device_id`, `target_ref_type`, `target_ref_id`, and `deep_link` for a future provider worker. It does not expose recipient account id, outbox status, activation codes, account secrets, prompt text, transcript text, biometric material, or requested task capability.
- `ProductNotificationOutboxService.markProviderSent(...)` and `markProviderFailed(...)` update only still-pending outbox rows by id, so a future provider worker can feed `sent` and `failed` lifecycle state without mutating resolved/cancelled rows.
- Pending device transfer outbox rows are marked `resolved` when the target accepts the transfer and `cancelled` when the source cancels it, keeping the pre-send queue aligned with local product state.

## Validation

- `DeviceTransferServiceImplTest` covers:
  - creating pending transfer requests
  - rejecting deleted account tombstones before transfer binding lookup or mutation
  - accepting transfer and moving binding
  - clearing old members and voiceprints
  - clearing the old primary session
  - notifying DeviceServer to clear voiceprint cache
  - cancelling pending requests without binding/cache side effects
  - listing pending incoming transfers
- `AppV2ControllerTest` covers transfer endpoint delegation.
- `DeviceServerMotionGatewayImplTest` covers voiceprint cache clear HTTP call.
- `ProductNotificationOutboxServiceImplTest` covers safe pending notification outbox persistence.
- `tests/ci/test_manager_api_device_transfer.py` statically guards API, table, service, and DeviceServer cache contracts.
- `tests/ci/test_manager_mobile_device_info.py` statically guards manager-mobile transfer API wrappers, device-detail entry, and device-list pending incoming entry.
- `tests/ci/test_manager_mobile_device_info.py` also guards the product notification runbook terms for pending incoming transfers.
- `tests/ci/test_manager_api_device_transfer.py` guards the notification outbox table, safe deep link contract, and non-sensitive event payload terms.

## Limitations

- Pending incoming transfer notification is currently pull-based in the device list page. It updates a best-effort native tabBar badge after pending transfer refreshes and reapplies the stored local badge count on app `onShow`; a local pending notification outbox plus safe provider payload builder exist for future push workers. Platform push notifications, native tabBar rendering on every platform, and background reminders are not proven locally. If push or reminders are deferred for release, the release ticket must record the deferral owner, due date, fallback path, and risk acceptance.
