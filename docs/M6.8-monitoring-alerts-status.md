# M6.8 Monitoring Alerts Status

Status: implemented as a minimal no-new-dependency monitoring slice.

Evidence Boundary:

This status is local/design-time evidence only. It is not a completion declaration and does not replace deployed scrape health, redacted secret mount evidence, dashboard rendering, alert firing, or DingTalk/WeCom webhook delivery evidence.

Implemented:

- Manager API exposes Prometheus text metrics at `GET /xiaozhi/internal/v1/metrics`.
- The metrics endpoint is protected by the existing Manager API server-secret filter; Prometheus reads the Bearer token from `/etc/prometheus/secrets/manager-api.secret`.
- Metrics include:
  - `biz_task_failure_ratio`
  - `dev_device_offline_total`
  - `u8_ota_failure_ratio`
  - `dev_device_rma_in_progress_total`
  - `dev_device_returned_total`
  - `dev_device_disposed_total`
  - `product_notification_pending_total`
  - `product_notification_failed_total`
- Added Prometheus scrape config and rules under `ops/monitoring`.
- Added Alertmanager webhook placeholders for DingTalk and WeCom.
- Added Grafana datasource and runtime dashboard provisioning, including task, offline, OTA, device lifecycle count, and product notification outbox panels.
- Added `ops/monitoring/docker-compose.yml` as a local/staging compose entry for Prometheus, Alertmanager, and Grafana, with `MANAGER_API_SECRET_FILE` support for the Manager API scrape secret.
- Added operations runbook `ops/runbooks/m6-monitoring-alerts.md` for pre-deployment checks, endpoint reachability, webhook secret handling, alert drills, screenshots/log evidence, and triage.

Alert rules:

- `ManagerApiTaskFailureRateHigh`: task failure ratio above 5% for 5 minutes.
- `DeviceOfflineSpike`: bound devices offline for 5 minutes.
- `OtaRolloutFailureRateHigh`: OTA failure ratio above 3% for 2 minutes.

Validation:

- Java unit coverage: `MonitoringMetricsServiceImplTest`, `MonitoringMetricsControllerTest`.
- Static CI coverage: `tests/ci/test_monitoring_alerts.py`.
- Runbook coverage: `tests/ci/test_monitoring_alerts.py` guards deployment steps, alert drill requirements, webhook placeholders, and local limitations.

Limitations:

- This slice avoids Spring Actuator/Micrometer to keep local builds offline-friendly; metrics are emitted by a small internal Prometheus text endpoint.
- Product notification outbox metrics expose local pre-send queue state only. Failed rows are expected to be marked by future push delivery code through `ProductNotificationOutboxService.markProviderFailed(...)`; this still does not prove platform push delivery, native tabBar rendering, or background reminders.
- Alertmanager webhook URLs are placeholders and must be supplied by deployment secrets.
- The Manager API scrape secret file is not committed; local/staging operators must create it or point `MANAGER_API_SECRET_FILE` at a deployment-managed secret file before starting Prometheus.
- Grafana/Prometheus/Alertmanager are not started in local CI; CI verifies the config artifacts, compose entry, alert contracts, and runbook statically.
- OTA rollout counters are aggregated across firmware releases with `active`, `published`, or `paused` status; retired and draft releases are excluded from the local alert ratio.
- Staging or production evidence is still required for actual scrape health, dashboard rendering, and DingTalk/WeCom webhook delivery.
