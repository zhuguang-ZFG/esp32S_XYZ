# 编码任务索引 v2

本文件把 [实施计划-v2.md](./实施计划-v2.md) 的 `M1 ~ M6` 子步骤映射到当前仓库的真实目录、主要文件、输入输出和验收命令。

用途：

- 把计划直接转成可派发的编码任务
- 降低“知道要做什么，但不知道改哪里”的切换成本
- 让每个子步都能独立验收

约束：

- 以 `docs/架构定稿-v2.md` 为语义基线
- 以 `docs/实施计划-v2.md` 为顺序基线
- 若文档与代码现实不符，先改文档，再改实现

---

## 0. 仓库角色映射

| 角色 | 目录 | 说明 |
|---|---|---|
| `U1` | `firmware/u1-grbl/Grbl_Esp32/` | 运动执行、私有协议解析、安全裁决 |
| `U8` | `firmware/u8-xiaozhi/` | 设备侧 AI MCU、U1 协议桥 |
| `BusinessServer` | `server/xiaozhi-esp32-server/main/manager-api/` | 账号、任务、资源、授权、持久化 |
| `DeviceServer` | `server/xiaozhi-esp32-server/main/xiaozhi-server/` | 设备会话、语音链路、任务转发 |
| `Client` | `server/xiaozhi-esp32-server/main/manager-mobile/` | 微信小程序 |

## 任务派发建议

适合直接派发给编码 agent 的最小单位：

1. 单文件或单模块变更
2. 单一输入输出契约
3. 单一验收命令
4. 不跨两个以上角色目录

建议优先派发顺序：

1. `M1.2 ~ M1.8`
2. `M2.1 ~ M2.8`
3. `M2.9 ~ M2.12`
4. `M3.*`
5. `M4.*`
6. `M5.*`
7. `M6.*`

不建议一次性派发的打包任务：

- `M2` 全部一起做
- `M3 + M4` 一起做
- `OTA + 配网 + 自检` 一起做
- `合规 + 运营 + 售后` 一起做
