# M5.1 provisioning status

Date: 2026-05-15

Status: design-time provisioning contract is in place.

Implemented scope:

- U8 default config enables `CONFIG_USE_ESP_BLUFI_WIFI_PROVISIONING=y` as the BLE/Blufi primary provisioning path.
- U8 default config keeps `CONFIG_USE_HOTSPOT_WIFI_PROVISIONING=y` as the SoftAP HTTP fallback path.
- `provisioning_contract.h` records the shared channel names, SoftAP endpoints, NVS credential keys, and BLE Just Works policy used by M5.1/M5.2/M5.3.
- `WifiBoard::StartWifiConfigMode()` starts Blufi before SoftAP so BLE can advertise before the fallback AP is exposed.
- The mini program device-config page now exposes `ble_blufi` as the primary mode and `softap_http` as the fallback mode.
- `blufi-config.vue` implements the design-time BLE client flow: open adapter, discover `Xiaozhi-Blufi` / `BLUFI_DEVICE`, connect, enumerate services and characteristics, enable notify, and write the provisioning payload through the shared contract.
- Static CI covers the default config, firmware/mobile provisioning contract fields, config-mode startup order, BLE client API usage, and primary/fallback mode wiring.

Still not complete:

- Real phone-to-U8 BLE provisioning test.
- Production Espressif BluFi frame compatibility, encryption handshake, and real-device response handling.
- Runtime resolution of Blufi and SoftAP coexistence beyond design-time startup ordering.
- NVS encryption enforcement and verification for Wi-Fi credentials, `device_secret`, and `server_host`.
- AP fallback end-to-end validation after BLE failure.

## Provisioning Evidence Gap Record

Use this structure when BLE/BluFi provisioning evidence is missing for a release candidate. It is not evidence of real phone-to-U8 provisioning.

- missing evidence scope: phone-to-U8 BLE provisioning, BluFi frame compatibility, encryption handshake, real-device response handling, BLE/SoftAP coexistence, NVS encrypted credential persistence, or AP fallback after BLE failure
- environment: staging device, release-candidate firmware, physical phone model, OS version, and WeChat version
- owner: named firmware or mobile owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep SoftAP fallback available, keep provisioning labeled design-time until hardware evidence is attached, and block release if neither BLE nor SoftAP has a passing real-device path
- risk acceptance: firmware owner, mobile owner, and release manager approval reference
- rollback trigger: BLE scan/connect/write fails, BluFi handshake fails, SoftAP fails after BLE failure, credentials are not persisted, or plaintext credentials are found in storage evidence
- follow-up evidence: firmware build id, phone model, OS version, WeChat version, BLE scan/connect/write logs, BluFi response logs, SoftAP fallback logs, NVS encrypted persistence proof, and redacted screenshots or serial logs
