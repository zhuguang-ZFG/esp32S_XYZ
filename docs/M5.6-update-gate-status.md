# M5.6 update gate status

Date: 2026-05-15

Scope implemented:
- Manager API rejects new task submission before task persistence or forwarding when device `status` or `runtime_state` is `UPDATING` or `UPGRADING`.
- The rejection uses the app-v2 safety/business error code `E_DEVICE_UPDATING`.
- U8 rejects incoming `motion_task` while `Application` is in `kDeviceStateUpgrading`.
- U8 emits a terminal `motion_event` with `phase:"failed"`, `error_code:"E_DEVICE_UPDATING"`, and the original task metadata when available.

Checks added:
- Java unit coverage for manager-api task submission rejecting an updating device before entitlement checks, insert, or forward.
- Static CI checks for manager-api update gate placement and firmware-side `motion_task` rejection.

Not verified in this slice:
- Real hardware OTA timing under WSS task traffic.
- End-to-end propagation through a live xiaozhi-server deployment.

## Update Gate Evidence Gap Record

Use this structure when update-gate evidence is missing for a release candidate. It is not evidence of real OTA/task contention behavior.

- missing evidence scope: real hardware OTA timing under WSS task traffic, live xiaozhi-server propagation, manager-api rejection before task persistence, U8 `motion_task` rejection while upgrading, or terminal `phase:"failed"` delivery with `E_DEVICE_UPDATING`
- environment: staging BusinessServer, xiaozhi-server, U8 device, release-candidate firmware, active WSS session, task source, and OTA release id
- owner: named backend, firmware, or operations owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: pause OTA rollout while active task traffic is unverified, keep manual task submission disabled during upgrade windows, and block release if either backend or firmware gate is bypassed
- risk acceptance: backend owner, firmware owner, operations owner, and release manager approval reference
- rollback trigger: manager-api persists or forwards a task while the device is updating, U8 accepts a `motion_task` in `kDeviceStateUpgrading`, terminal failure event is missing, or live xiaozhi-server does not propagate `E_DEVICE_UPDATING`
- follow-up evidence: OTA timing log, WSS task traffic log, manager-api rejection log, database proof that no task was persisted, U8 serial rejection log, motion-event payload with `phase:"failed"`, and live xiaozhi-server forwarding trace
