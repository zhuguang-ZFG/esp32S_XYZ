# M5.2 AP fallback provisioning status

Date: 2026-05-15

Status: design-time mini program AP fallback contract is in place.

Implemented scope:

- Mini program `pages/device-config/index` remains registered in `pages.json` and keeps Wi-Fi provisioning as the default mode.
- `provisioning-contract.ts` centralizes the SoftAP HTTP fallback endpoint contract:
  - base URL: `http://192.168.4.1`
  - scan: `GET /scan`
  - submit: `POST /submit`
  - exit: `POST /exit`
  - submit payload fields: `ssid`, `password`
- `wifi-selector.vue` uses the shared contract to scan AP networks and accepts the known response shapes: `{ success, networks }`, `{ aps }`, and legacy array payloads.
- `wifi-config.vue` uses the shared contract to check connectivity, submit credentials, and call exit after a successful submit.
- Static CI covers the page registration, component wiring, endpoint contract, submit payload, scan compatibility, and fallback user prompt.

Still not complete:

- Real phone connected to U8 SoftAP and successful credential submit.
- AP fallback after an actual BLE failure.
- U8-side runtime verification that `/scan`, `/submit`, and `/exit` remain compatible after BLE/SoftAP coexistence changes.
- NVS encrypted persistence verification after AP fallback provisioning.

## AP Fallback Evidence Gap Record

Use this structure when SoftAP fallback evidence is missing for a release candidate. It is not evidence of a real phone-to-U8 SoftAP provisioning path.

- missing evidence scope: phone joins U8 SoftAP, scan response, credential submit, exit request, AP fallback after BLE failure, U8 endpoint compatibility, or NVS encrypted persistence after SoftAP provisioning
- environment: staging device, release-candidate firmware, physical phone model, OS version, and WeChat version
- owner: named firmware or mobile owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep manual Wi-Fi retry instructions visible, keep BLE provisioning available, and block release if SoftAP is the only remaining provisioning path and lacks evidence
- risk acceptance: firmware owner, mobile owner, and release manager approval reference
- rollback trigger: phone cannot join SoftAP, `/scan` shape is incompatible, `/submit` fails, `/exit` fails, BLE failure does not expose fallback, or credentials are not persisted
- follow-up evidence: firmware build id, phone model, OS version, WeChat version, SoftAP association screenshot, `/scan` response, `/submit` request/response, `/exit` result, BLE-failure-to-SoftAP screen recording, and NVS persistence proof
