# M5.5 OTA backend status

## Scope

This slice adds the design-time backend domain for OTA release planning in `manager-api`.

## Implemented

- Added `firmware_releases` Liquibase migration and registered it in `db.changelog-master.yaml`.
- Added `V2FirmwareReleaseEntity` and `V2FirmwareReleaseDao`.
- Added `FirmwareReleaseService` with:
  - `publishDevRelease(...)` for publishing one dev-channel release plan.
  - release metadata validation aligned with M5.4 clients: HTTPS URL, 64-character lowercase hex SHA-256, and base64 signature.
  - `findUpgradeForDevice(...)` for selecting the latest published release by channel.
  - latest published release selection is deterministic: `published_at` descending, then `release_id` descending as a tie-breaker.
  - device rollout selection using a deterministic SHA-256 bucket of `device_id`.
  - rollout steps clamped to 10 / 50 / 100.
  - `recordInstallResult(...)` and failure-rate auto-pause.
- Added minimal admin REST surface:
  - `POST /admin/firmware-releases` publishes a dev release through `FirmwareReleaseService.publishDevRelease(...)`.
  - `POST /admin/firmware-releases/{releaseId}/install-result` ingests device install success/failure and returns the updated release status.
  - Both endpoints require `sys:role:superAdmin`.
- Added an internal DeviceServer bridge:
  - `POST /internal/v1/firmware/upgrade-plan` is protected by the existing DeviceServer bearer token.
  - `POST /internal/v1/firmware/install-result` lets DeviceServer report OTA install success/failure without using the admin API.
  - The endpoint rejects disposed devices before returning a plan.
  - `xiaozhi-server` OTA handling can call BusinessServer with `device_id`, channel, and current firmware version.
  - `xiaozhi-server` treats BusinessServer `Result.code != 0` as no usable OTA plan, even when HTTP status is 200.
  - `xiaozhi-server` revalidates BusinessServer plans against the U8 client contract before applying them: HTTPS URL, 64-character lowercase SHA-256, and base64 signature.
  - Signed BusinessServer plans populate OTA `firmware.release_id`, `firmware.version`, `firmware.url`, `firmware.sha256`, and `firmware.signature`.
  - `xiaozhi-server` exposes `POST /xiaozhi/ota/install-result` as a device-facing result relay and forwards to BusinessServer with the same internal token. BusinessServer `Result.code != 0` makes the relay report `forwarded = false`.
  - U8 firmware has a design-time install-result path: persist pending `release_id`, report local upgrade failure, and report success after post-reboot OTA validation.
  - Existing `data/bin` local scanning remains as a development fallback when no BusinessServer plan is configured or returned.
- Static CI covers the changelog, table columns, entity/DAO mapping, dev publish path, REST publish/result-ingest surface, rollout logic, and failure-rate pause contract.
- Java unit coverage verifies that publish rejects release metadata the U8 client cannot validate.
- Python unit coverage verifies the DeviceServer OTA bridge request shape and signed metadata mapping.
- M6.8 local monitoring reads `firmware_releases.install_count` / `failure_count` into `u8_ota_failure_ratio`, and the local alert rule watches that metric.

## Not yet complete

- Real deployed DeviceServer / U8 evidence that a live device consumes a BusinessServer release plan.
- Real U8 firmware result POST after reboot / rollback validation on hardware; the software path exists, but delivery is not proven on a device.
- Real deployed Prometheus scrape health, alert firing, and DingTalk/WeCom webhook delivery for M5.5 failure-rate alerts.
- Real single-device OTA plan hit from a live device.
- Production signing private-key custody, OTA public-key provisioning, and live signature evidence remain part of the OTA hardening path.

## OTA Backend Evidence Gap Record

Use this structure when OTA backend evidence is missing for a release candidate. It is not evidence of a deployed BusinessServer OTA rollout.

- missing evidence scope: live BusinessServer release-plan consumption, live DeviceServer / U8 upgrade-plan request, deployed install-result relay, rollback validation on hardware, deployed Prometheus scrape health, alert firing, DingTalk or WeCom webhook delivery, single-device plan hit, signing private-key custody, OTA public-key provisioning, or live signature validation
- environment: staging or production BusinessServer, DeviceServer, U8 device id, firmware release id, rollout channel, OTA host, monitoring stack, and webhook target
- owner: named backend, firmware, security, or operations owner accountable for closing the gap
- due date: calendar date for closing the missing evidence
- fallback path: keep BusinessServer OTA rollout disabled or dev-channel only, keep `data/bin` fallback limited to development, pause rollout on missing monitoring, and block production if live plan consumption or install-result delivery is unproven
- risk acceptance: backend owner, firmware owner, security owner, operations owner, and release manager approval reference
- rollback trigger: live device cannot consume a BusinessServer plan, DeviceServer accepts invalid metadata, install-result relay fails, rollback validation fails, failure-rate alert does not fire, webhook delivery is missing, or signing custody / public-key provisioning is unresolved
- follow-up evidence: release id, signed metadata record, DeviceServer upgrade-plan request/response, U8 serial log, BusinessServer install-result row or log, rollback drill evidence, Prometheus target screenshot, alert firing screenshot, DingTalk/WeCom delivery screenshot, signing custody record, and public-key provisioning record

## Notes

This remains a design-time bridge, not a full rollout console. It creates a stable storage and service boundary for M5.4 clients and later M5.6 task gate integration. The local `data/bin` fallback is intentionally retained for development, but production OTA should publish signed metadata through BusinessServer.
