### 3.12 M2.3 motion_task 转发同步阻塞 submit_task 事务（新发现，未修）

- 文件：`appv2/service/impl/AppV2ServiceImpl.java::submitTask`（外部 linter 落入 Line 195 调用 `deviceServerMotionGateway.forwardAcceptedTask`）+ `DeviceServerMotionGatewayImpl.java`
- 现状：
  - `forwardAcceptedTask` 在 `submitTask` 的 `@Transactional` 范围内同步发 HTTP 给 DeviceServer
  - DeviceServer 慢 / 离线时 `RestClientException → RenException` 直接外抛 → 外层事务回滚
  - 等于"DeviceServer 离线就不能落任务"
- 与 v2 偏差：
  - §12bis 设备掉线策略要求"任务先落库 + `device_offline` 事件"；当前实现直接拒绝
  - §6.1bis 幂等要求 `(account, device, request_id)` 重复请求返回首次 `task_id`；当前事务回滚后第二次请求会再次失败而非走幂等
  - §17 联调顺序也明确 Edge-B 是"内部事件总线 / 异步"，不应把外部 RPC 塞进数据库事务
- 修复方向：
  - 任务落库与转发拆开：先 commit 事务再异步发 motion_task，或加重试队列
  - DeviceServer 不可达不抛错，落到 `task.status = 'pending_dispatch'`，由后台轮询补送
  - 失败事件走 `motion_event` 上行链路而不是 `submitTask` 的同步返回
- 优先级：M2.3 真实联调前必须解决；本报告记录但本轮不动该代码（按 §1，不属于本批次范围）

### 3.13 ESTOP 自动抬笔（待 M0f 实物验证后再修）

- 与 §2.5 同一问题：`Protocol.cpp:351` ESTOP 路径仅 `mc_reset()`，无显式 Z 抬笔
- 当前 PCB 没有独立抬笔舵机，Z 抬笔即步进运动；硬件未到位前盲改 `mc_reset` 路径风险大
- 待 M0f 实物 + Grbl_Esp32 真实 Z 行为核实后再决定：
  - 方案 A：U1 在 ESTOP 入口先发 `G91 G0 Z<pen_clearance>` 再 `mc_reset`（但 ESTOP 应是"立即停"，再发运动指令逻辑上矛盾）
  - 方案 B：依赖 §10bis.7 让 BusinessServer 在 STOP 受控停止时抬笔；ESTOP 视为"机械可能压笔但电源仍在"的极端状态，由用户硬切断
  - 方案 B 更符合 ESTOP 语义，建议在 v2 §10bis.12 / §14bis.2 显式登记"ESTOP 不保证抬笔，请配合断电"
