### 2.5 M1.5 U1 PAUSE / RESUME / STOP / ESTOP — ⚠ 已实现但有两处偏差

代码：`Protocol.cpp:316-360`

- ✓ PAUSE → `feedHold` + `ack`
- ✓ RESUME → `cycleStart` + `ack`
- ✓ STOP → `private_protocol_begin_controlled_stop()` 后 `ack`，失败返回 error（语义符合 §14bis.5 / §15.4 不伪造 cancelled 的纪律）
- ⚠ ESTOP 实现：
  ```cpp
  sys.is_homed = false;
  sys.last_alarm = ExecAlarm::EmergencyStop;
  sys_rt_exec_alarm = ExecAlarm::EmergencyStop;
  send_private_protocol_ack_with_state(...);
  mc_reset();
  ```
  - 立即 ack ✓（符合 v2 §15.4 ESTOP 高优先级语义）

## 3. 与文档不符 / 需要补登记的代码质量问题

### 3.1 `Report.cpp:857` 三元两端同值（已修，2026-05-14）

- 修前：`const char* error_code = sys.state == State::Alarm ? "null" : "null";`
- 修后：`const char* error_code = "null";`，并加注释指明 status 帧 error_code 恒为 null，错误码走独立 error 帧。
- 后续（M0a 推进时）：把"status 帧 error_code 永远 null"作为契约写进 `docs/schemas/edge_d/get_status.status.schema.json`。

### 3.2 `RunPath` 单段超时 150 ms 风险

- `dlc_motor_control_p1_ai_board.cc:298`：每个 PATH_SEG 等待 ack 的超时仅 150 ms
- 若 U1 内部正在 `feedHold` 或忙于上一段，150 ms 不够稳定
- v2 §15.4 没规定单段 ack 时延上限，但 M1.10 / M2.5 实测前需要把"段内超时"作为可调参数

### 3.3 PATH_END 同步阻塞 120 s

- `dlc_motor_control_p1_ai_board.cc:308`：PATH_END 阻塞等响应最长 120 000 ms
- v2 §16.5 第 4 项要求"PATH_END 改异步执行：执行期间按段产生 progress 事件"
- 当前实现是单次同步等结果，**未实现 progress 流**
- M2.12（v2 §17.6 run_path 端到端）依赖此项，必须在 M2 阶段补
- 影响：长任务（写一首诗）会让 U8 一根线程阻塞 120 s，期间无 progress 上报，小程序看到的是"空跑 + 突然 done"

### 3.4 task_id 全链路一致性未保证

- 多处出现写死的本地 task_id：
  - U8 `RunPath`: `"t_path_local"`
  - U8 `self.motor.home` MCP: `"t_home_local"`
  - U8 `self.motor.move_abs`: 由 `SendU1MoveCommand` 生成
- 这些是 MCP 调试入口的快捷值，**M2.3 真实链路下必须由 BusinessServer 生成 UUID 透传到 U1**
- 当前未在 §16.4 实施清单里列入"任务 ID 透传改造"，建议把它登记为 M2.5 的子项

### 3.5 文档保留期已对齐（撤销）

§10bis.11 / §10ter.6 / §22.5 三处审计保留期已对齐到 180 天（详见 §1.5）。本报告早期 draft 误判，已撤回。

### 3.6 server 端 Edge-B 完全没动

- `xiaozhi-esp32-server/main/xiaozhi-server/core/handle/` 没有 `motionHandle.py`
- `xiaozhi-esp32-server/main/manager-api/src/main/resources/db/changelog/` 一堆迁移但没有 v2 §16.2 第 1 项要求的：
  - `accounts / devices / device_bindings / activation_codes / tasks / fonts / copybooks / assets / entitlements / voiceprints / members / ai_providers / ai_plans / firmware_releases / device_supplies / diagnostic_reports / safety_audit / content_audit`
- **manager-api 当前是上游 xiaozhi-server 的"账号/Agent 管理"语义，与 v2 BusinessServer 语义不重合**
- 影响：M2 整段都是新建工作，不是现有改造

### 3.7 Client 完全未按 v2 改造

  - v2 §8 客户端 capability 任何一个对应页面（绑定 / 转赠 / submit_write_text / submit_draw / 实时订阅）
- 影响：M2.8 "三页最小可用" 是从零开始

### 3.8 xiaozhi-server 现有意图链路与 §6bis.3 不接

- xiaozhi-server 现有 `core/providers/intent/{intent_llm, function_call, nointent}` 三种模式（详见上一轮分析），对应 §6bis.1 的意图分流
- **但**：
  - `POST /internal/v1/intent_submit` 这个 §6bis.3 接口在 manager-api 不存在
  - 当前路径走 `tools/server_mcp` / `tools/device_mcp`，绕过 BusinessServer 鉴权层
  - 这是 v2 §6bis.5 边界纪律明确禁止的"DeviceServer 不调用 BusinessServer 之外的写路径"
- 影响：M4.3 / M4.4 必须先做"绕回 BusinessServer 的反向接口"

### 3.9 `/api/v1/login` 信任客户端 unionid（红线，未修）

- 文件：`server/xiaozhi-esp32-server/main/manager-api/src/main/java/xiaozhi/modules/appv2/service/impl/AppV2ServiceImpl.java::login()`
- 同时：`security/config/ShiroConfig.java:85` 把 `/api/v1/login` 标为 `anon`
- 现状：login 接口直接读取 request body 里的 `unionid`，未做任何来源校验，等价 **"输入任意 unionid 即可登录"**
- 攻击面：任何匿名调用方都可登录已绑定的 v2 账号 → 拿到 JWT → 后续所有 v2 接口都"以受害者身份"被调用
- 与 v2 §11.1 冲突：文档要求 `client.login(code) → BusinessServer 换取 openid/unionid`，必须服务端调微信 `jscode2session`，禁止信任客户端直传
- 修复方向：
  - 入参由 `unionid` 改为 `code`
  - 服务端配置微信 AppID/AppSecret，调 `jscode2session` 换得真 unionid
  - dev 环境可走"测试 unionid + 服务端白名单"开关，但默认关闭
  - 接口仍可保 anon，因为登录前没有 JWT；安全保障来自微信换取链路而非 Shiro 过滤器
- 阻塞：M2.2 不能算完成，M2.3 起的 Edge-B 鉴权设计也将被此漏洞污染

### 3.10 v2 账号底层共用弱密码（红线，未修）

- 文件：`AppV2ServiceImpl.java:194 generateStrongPassword()` 返回常量 `"V2Temp!Pass1"`
- 当前实现把每个 v2 账号在 `sys_user` 表写死同一个密码
- 攻击面：知道账号 username（格式固定 `wx:<unionid>`，且 unionid 可枚举/泄漏）+ 已知默认密码 → 走旧 `/user/login` 接口冒用 v2 账号
- 修复方向：
  - 改用 `SecureRandom` 生成账号级随机密码，仅写入数据库不向外回显
  - 或彻底拆分鉴权链：v2 账号不复用 `sys_user`，独立 token store
- 阻塞：与 §3.9 同等优先级，M2.2 红线之一

### 3.11 v2 JWT 载荷缺 `primary_session_id`（功能性偏差，未修）

- 文件：`AppV2ServiceImpl.java:93-97`，复用 `SysUserTokenService.createToken(userId)` 直发 token
- 与 v2 §11.1 冲突：JWT 载荷应含 `account_id + issued_at + expires_at + primary_session_id`，缺最后一项无法支撑 §12.2 "主控会话仲裁"
- 影响：M6.4 主控仲裁里程碑需要回填，建议在 §3.9 修复时一并扩展 token 字段
- 不构成红线，但是 M2.2 完成线的硬约束
